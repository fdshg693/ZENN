## はじめに

Claude Code を本格的に使い込むと、すぐにぶつかる疑問がある。

> **「この指示は、プロンプトに書くべきか？ `AGENTS.md` に書くべきか？ スキルにすべきか？ それとも専用エージェントにすべきか？」**

どれに書いても「一応動く」のが厄介なところで、動くからこそ場当たり的に積み上がっていき、いつの間にかメンテナンス不能な設定群になる。実装者ごとにスタイルが違えば、チームで共有したときにさらに事故が起きる。

本記事では、スキルやエージェントファイルの**書き方そのもの**（フロントマターの書式、ツール指定の構文など）には立ち入らない。そこは公式ドキュメントに譲る。代わりに、**「何をどこに書くか」というレイヤー設計の話**に絞って、できる限り一般論として通用する形で整理する。Claude Code に限らず、同種の設定機構を持つエージェントツール（Cursor、GitHub Copilot、Cline、将来登場する別製品）にも、そのまま読み替えられるはずだ。

もっとも、実際には製品ごとに細部はかなり違う。どのファイルが自動ロードされるか、パス条件でどこまでスコープを切れるか、スキルとエージェントの境界をどう設計しているか、プラグインやマーケットプレイスを持つか、といった仕様は揃っていない。したがって本記事は**レイヤー設計の考え方自体は一般化しつつ、具体例と評価軸は主として Claude Code を対象に述べる**。

論拠として、Anthropic 自身の設計思想（[Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)、[Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）、および長文コンテキストにおける LLM 挙動の研究（Liu et al., "Lost in the Middle", 2023 など）、ソフトウェア工学の古典的原則（単一責任の原則、関心の分離）に繰り返し言及する。

:::message
対象読者は「スキルや Subagent の基本は触ったことがある」方。概念の紹介は省き、**使い分けの判断基準**に焦点を当てる。
:::

## 前提：コンテキストは有限で、かつ「長ければ長いほど良い」わけではない

設定の置き場所を議論する前に、共有しておきたい前提が一つある。

LLM は長いコンテキストほど情報を見落としやすい。Liu らの "Lost in the Middle"（2023）は、入力中盤に置かれた情報がモデルに無視されやすいことを実験的に示した。Anthropic も自社ブログ ["Effective Context Engineering"](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) の中で、"context rot"（コンテキストが長くなるほど指示追従性や推論の質が劣化する現象）に言及し、**「すべてを詰め込む」のではなく「必要なものだけ流し込む」設計** を推奨している。

つまり、設定をどこに書くかという問題は、本質的に **「いつ、どの情報を、どれだけの優先度でモデルに渡すか」** の設計問題であり、単なるファイル配置の好みではない。この視点を踏まえて、以下の各機構の特性を見ていく。

## プロンプト・RULES ファイル・スキルの比較

まず、常時コンテキストに乗せる系統（プロンプト／RULES ファイル）と、必要に応じて読み込む系統（スキル）を対比する。

ここで言う **RULES ファイル** とは、**「プロンプトでもスキルでもなく、エージェントに自動ロードされる持続的な指示レイヤー」** の総称である。ツールごとに実装は異なる。たとえば Cursor なら `.cursor/rules/*.mdc`、Cline なら `.clinerules/`、Claude Code なら `CLAUDE.md` 系ファイルと `.claude/rules/` がこのレイヤーに相当する。

ただし **Claude Code の実装上は `AGENTS.md` は直接は読まれず、読むのは `CLAUDE.md` / `CLAUDE.local.md` / `.claude/rules/**/*.md`** である。[AGENTS.md](https://code.claude.com/docs/en/memory#agents-md)既存の `AGENTS.md` を使い回したい場合は、公式ドキュメントのとおり `CLAUDE.md` から `@AGENTS.md` と import して取り込む。本記事ではレイヤー設計の比較を簡潔にするため、こうした Claude Code の `CLAUDE.md` 系ファイルと `.claude/rules/` をまとめて「RULES レイヤー」と呼ぶ。

### プロンプトの弱点：静的で巨大になりがち

プロンプトは最も原始的な指示手段だが、エージェント運用の観点では以下の弱点がある。

- **1 ファイルに集約するしかない**。分割参照は可能でも、タスク開始時点では「どの参照先が必要か」をモデル側で判断できない。結局、保険で全部読ませることになり、肥大化する。
- **動的取得ができない**。たとえば「タスク完了後にコミットさせたい」場合、プロンプトにコミット手順の説明を含めておく必要がある。タスク開始前からコンテキストを食い、さらに本題と混ざって注意力を分散させる。
- **Claude Code には「プロンプトファイル」という概念がない**。GitHub Copilot の `.github/prompts/*.prompt.md` のように、チャットから特定プロンプトを呼び出す仕組みが存在しない。定型プロンプトを使い回したい場合は、**スキルの手動実行で代替するのが現実解** となる。`/skill-name` 形式でユーザー側から明示的に呼び出せば、「必要なときだけ読み込ませる」という目的はスキルでも十分満たせる。
   - **モデルが勝手に呼び出すのを避けたい場合**も、スキルのフロントマターに `disable-model-invocation: true` を指定すればよい。これによりユーザー呼び出し専用になり、description が常時コンテキストに載るコストも抑えられる。

### Claude Code における `CLAUDE.md` 系ファイルと RULES の関係

Claude Code では、このレイヤーには実体として **`CLAUDE.md` 系ファイル** と **`.claude/rules/`** の 2 種類がある。

- `CLAUDE.md` / `.claude/CLAUDE.md` / `CLAUDE.local.md` は、**配置場所によってスコープと読み込みタイミングが決まる持続指示**。
- `.claude/rules/` は、それを**より細かく分割・条件付き適用できる一般化された仕組み**。
- `AGENTS.md` は Claude Code では直接読まれないため、必要なら `CLAUDE.md` から import して取り込む。

要するに、Claude Code では RULES ファイルが `CLAUDE.md` 系ファイルを包含する、くらいの理解で十分である。以後は、特に区別が必要な場面を除いて **「RULES レイヤー」** とだけ書く。

`CLAUDE.md` 系ファイルは、原則として**現在の作業ディレクトリから親方向にたどって見つかったものは起動時に読み込まれ、サブディレクトリ配下のものはそのディレクトリのファイルを読んだときに遅延ロードされる**。この「配置で読み込みタイミングを制御する」性質が、長所にも短所にもなる。

**短所：適用範囲が広すぎる**

- 書ける内容は**汎用的なもの**に限定される（コーディング規約、言語／フレームワーク、ビルド手順など）。
- 「レビュー指摘への対応手順」「リリース時のチェックリスト」のような**局所的な知識**を書き込むと、関係ない会話でも常に読まされて、コンテキストを無駄に消費する。
- プロジェクトごとに内容が大きく変わるため、**汎用 RULES は原理的に作れない**。ベストプラクティス集のようなものは成立しにくい。

**長所：常時ロードだからこそ効く指示がある**

一方で、常時ロードされるという性質は裏を返せば **「絶対に読み飛ばされない」** ということでもあり、これは他の機構では代替しにくい。

- プロジェクト固有の制約（「このリポジトリではテストは `just test` で実行する」「DB マイグレーションは手で書かず `sqlc generate` を通す」）は、毎回読ませたほうが事故が起きにくい。
- **エージェントに必ず守らせたい個人ルール**（「回答は必ず日本語で」「コミットメッセージは Conventional Commits」「破壊的な操作の前に必ず確認を挟む」）は、スキル任せにすると発動漏れが致命傷になる。RULES に書いて常時ロードさせるのが最も堅い。

つまり RULES は「**汎用性と恒常性が高い最小限のルール**」に絞ると最もコストパフォーマンスが良い。

**配置ベースのスコープ制御はできるが、Claude Code では RULES がさらに一般化する**

`CLAUDE.md` はサブフォルダにも配置でき、**そのフォルダ配下のファイルにアクセスしたときに追加で読み込まれる**。モノレポの `backend/CLAUDE.md` と `frontend/CLAUDE.md` を使い分ければ、全会話で全部を読み込む無駄を避けられる。これだけでもコンテキスト消費はかなり抑えられる。

ただし Claude Code では、これは `.claude/rules/` でできることの一部に近い。`.claude/rules/` では、

- `paths` なしの rule は起動時に `.claude/CLAUDE.md` と同等の優先度で読み込まれる
- `paths` 付きの rule は、該当パターンに一致するファイルを Claude が読むときにだけ読み込まれる

という形で、**「常時ロード」と「条件付きロード」を 1 つの仕組みで扱える**。そのため Claude Code では、`CLAUDE.md` の配置による制御も重要だが、より包括的に語るなら `.claude/rules/` を中心に据えた方が整理しやすい。

**より柔軟な条件制御が欲しい場合：Claude Code でも `.claude/rules/` が使える**

Claude Code には現状この機構がない、という理解は正確ではない。実際には `.claude/rules/` により、**ファイル種別やサブディレクトリに応じた path-scoped rules** を定義できる。「`src/api/**/*.ts` を読むときだけ API ルールを読む」といった制御は、公式にサポートされている。

この意味で、Claude Code の `CLAUDE.md` 系ファイルは「配置ベースでスコープする基本形」、`.claude/rules/` は「その上位互換に近い整理機構」と捉えると分かりやすい。フォルダ配置による遅延読み込みを使いたい場面もあるが、**記事全体のレイヤー設計としては RULES の中に `CLAUDE.md` 系ファイルを含めて扱う**のが自然である。

**ただし限界もある：ファイル位置に基づく制御はできても、タスク進捗に応じた動的ロードはできない**

サブフォルダ配置の `CLAUDE.md` も、`paths` 付き `.claude/rules/` も、結局は **「どのファイルを触っているか」という静的情報** を主軸に切り替わる仕組みである。「調査フェーズが終わって実装フェーズに入ったから、ここでリリース手順の RULES をロードする」といった、**タスクの進行状態に応じた真の動的ロード**は苦手だ。ここを埋めるのが、次に述べるスキルの役割になる。

### スキルの優位性：Progressive Disclosure

スキルの本質は、**「必要になったときに、必要な分だけ読み込む」** 仕組みにある。これは UX 設計やドキュメンテーションの文脈で古くから言われている **Progressive Disclosure（段階的開示）** の原則と一致する。

| 観点                       | プロンプト                                               | RULES レイヤー | スキル                 |
| -------------------------- | -------------------------------------------------------- | ----------------------------------------- | ---------------------- |
| 動的ロード                 | ×                                                        | △（path-scoped rule やサブディレクトリ `CLAUDE.md` による条件付きロードは可能） | ○                      |
| ファイル分割               | △（参照のみ）                                            | ○（imports、複数 `CLAUDE.md`、`.claude/rules/`） | ○（フォルダ単位）      |
| 状況特化のルール           | △（開始時にユーザーが指定する必要があり、タスク推移への対応は不可） | △（ファイルパスベースまでは強い。進捗ベースは不可） | ○                      |
| 常時コスト                 | 中                                                       | 中〜大（常時 rule と条件付き rule の設計次第） | 小（メタデータのみ）   |
| プロジェクト横断の再利用   | △                                                        | △（ユーザー-level rules や symlink 共有は可能） | ○                      |
| 「絶対に読まれる」保証     | ○                                                        | ○（常時 rule）/ △（条件付き rule は対象ファイル読取時） | △（description 次第）  |

特に重要なのは次の 2 点。

1. **フォルダ単位で責務を分けられる**。単一責任の原則 (Single Responsibility Principle) を、設定ファイルのレベルでそのまま適用できる。スキル 1 つ ＝ 1 つのユースケースという粒度は、人間にとってもモデルにとっても理解しやすい。
2. **遅延読み込みにより、モデル自身が「このスキルを今読むべきか」を判断する**。プロンプトから他ファイルを参照させる方式では、「いつ参照するか」がモデル任せになるが、判断材料（description）が十分でないまま読み込みが起きがち。スキルは `description` フィールドで「このスキルが発動すべき状況」を明示できるため、判断材料が揃った状態で読み込まれて、同階層の追加ファイルの中のどれを読むか適切に選べる。

結果として、スキルは **「コンテキスト負荷を最小化しつつ、状況ごとの個別対応ができる」** という、プロンプトと RULES の中間点にあたるレイヤーを埋める。GitHub 操作用、レビュー用、リリース手順用といった **場面特化の知識** を、プロジェクトをまたいで共有できる汎用スキルとして切り出せる点は、プロンプト派・RULES 派では得られない強みだ。

### スキル化の判断基準と落とし穴

「作って損はないか？」という観点で言えば、**たまにしか使わない知識でもスキル化しておく価値はある**。スキル本体の内容は常時コンテキストに乗らないため、存在するだけのコストは小さい（description 分のメタデータのみ）。モデルが勝手に呼んでしまって困る場合は、`disable-model-invocation: true` や、フォルダ移動等による一時的な無効化で対処すればよい。

一方、スキル運用でやりがちな失敗もある。

- **「AI が適切なスキルを自動で呼ぶ」ことに依存しすぎる**。遅延読み込みは、裏を返せば「読まれない可能性がある」ということ。クリティカルな手順はスキル任せにせず、プロンプトや RULES レイヤーに明示的なフックを置いて発動確率を上げるべき。
- **スキル間の責務が重複している**。GitHub 操作用スキルが複数あると、モデルはどれを使うか判断できず、適切なものが呼ばれるとは限らない。description を明確にし、**スキル同士が排他的**になるように設計する。これは API 設計における「1 機能 1 エンドポイント」と同じ思想。

## エージェントファイルの使いどころ

スキルとエージェントファイル（Subagent 定義）の役割が一番混同されやすい。結論を先に言うと、**エージェントファイルの本質的な価値は「コンテキストの分離」と「ツールセットの分離」** にある。それ以外の用途はほぼスキルで代替可能だ。

ただし、「代替可能だから使う意味がない」ということではない。エージェントにスキルを読ませるより、エージェント定義ファイル本体に書いた方が、スキル呼び出しのオーバーヘッドを避けられ、かつシステムプロンプトに直接埋め込むことでより強い指示を与えられる、というメリットがある。

### スキルで代替できる領域

以下のケースは、原則スキルで十分。

- **短いタスクでの手順指示** → スキルで十分。
- **参照すべきドキュメントの明示** → スキルで十分。
- **単発のサブエージェント呼び出し** → 「このフォルダを読んでこう動け」と指示するスキル ＋ 汎用 Subagent の組み合わせで足りる。

ただし次のような場合は、スキルで代替可能でもエージェントファイルに書く方が適切なことがある。

- そのエージェント 1 人しか使わない内容で、わざわざスキルに切り出しても再利用されない
- 重要度が高く、システムプロンプトに埋め込んで確実に順守させたい

### エージェントファイルでしかできないこと

逆に、以下はエージェントファイルが強みを発揮する領域。

#### 1. 長時間・多段タスクのワークフロー固定

複雑なタスクはフェーズを踏む（調査 → 設計 → 実装 → テスト → PR 作成）。このような**常に従ってほしいワークフロー**は、スキルではなくエージェントのシステムプロンプト側に書くのが望ましい。スキルは「読んで使い終わる」性質が強く、タスク全体を通じて遵守させるのは苦手だからだ。

Anthropic の "Building Effective Agents" でも、エージェントパターンの一つとして **Orchestrator-Workers**（オーケストレータが全体を分解し、ワーカーに委譲する）が挙げられている。オーケストレータ側の「分解方針」はまさにワークフローであり、エージェントの土台に据えるべき情報だ。

#### 2. ツールセットの制御

これが最も決定的な差別化要因。

- メインエージェントには持たせたくない MCP サーバー／ツールがある
- 逆に、あるタスクにしか必要ないツールを常時コンテキストに乗せたくない

この要件を満たすには、ツールセットを絞ったエージェントファイルを作るしかない。特に **MCP サーバーは情報量が多く、description だけでコンテキストをそれなりに食う**ため、切り分けの価値が大きい。

#### 3. 日常使いのメインエージェントのカスタマイズ

- 「必ず Web 検索して最新情報を確認する」
- 「回答は必ず出典付きで返す」
- 「コード変更時は必ずテストを書く」

こうした**個人スタイル**は、エージェントファイルでデフォルトエージェントをラップし、それをメインにして使うと最もきれいに収まる。スキルに入れても発動条件の制御が難しいルールは、常時ロードされるエージェントのシステムプロンプトに置くべき。

#### 典型例

- **自分専用の説明エージェント**：実装や調査の後、学習になりそうな内容を詳細に解説してくれる。頻繁に勉強目的で使うならエージェント化してストレスなく毎回利用できる（Claude Code はメインエージェントの変更が CLI オプション経由になり少し使いづらい。GitHub Copilot や Cursor であればカスタムエージェントをメインとして簡単に切り替え可能）。たまにしか使わないなら、説明スキルとして呼び出せれば十分だ。
- **Linear 専用サブエージェント**：タスク管理は Linear だが、メインエージェントに Linear MCP の知識や運用ルールを常時持たせる必要はない。タスク取得・更新だけ Subagent に委譲し、メインは実装に集中できる。

## ベストプラクティス

### 1. メタスキルで設定ファイル自体のメンテナンスを効率化する

スキルやエージェントファイルの**フロントマター**（ツール指定など）は、そうそう変わらないし、人間がレビューすべき箇所なので、自動化の優先度は低い。

一方、本文に埋め込まれたプロンプト部分は量が多く、改善サイクルも早い。ここは**メタスキル**（スキルを書くためのスキル）を導入する価値がある。

- 文章を構造化（冗長表現の除去、見出しの正規化）
- 英語化による効率化（LLM は英語プロンプトで挙動が安定しやすい）
- 自分流のスタイル（「Why → How → Example の順で書く」など）の強制
- Claude Code の公式 Docs を Web ツールで取得させて、ベストプラクティス・新機能に沿って更新させる

ソフトウェア工学における "Don't Repeat Yourself" や "Convention over Configuration" の思想を、設定ファイル自体に適用すると考えるとわかりやすい。

### 2. スキルは SKILL.md を薄いエントリポイントにする

これは Anthropic 公式の推奨でもある Progressive Disclosure の実践形。

- `SKILL.md` には **「何のスキルか」「いつ呼ぶか」「どのサブファイルをどの条件で読むか」** だけを書く
- 詳細な手順・リファレンスは別ファイルに分割
- モデルは最初に `SKILL.md` だけ読み、必要に応じてサブファイルを選択的に読む

これにより、**汎用スキル**（様々なプロジェクト・状況で使い回すスキル）を作る際に、コンテキスト消費を最小限に抑えられる。100 行のスキルを全員が毎回ロードする設計より、20 行のエントリ ＋ 条件付きサブファイル参照の方が圧倒的にスケールする。

### 3. 複雑な手順はスクリプト化する

モデルに毎回同じ手順を踏ませるのは、**プロンプトを通じたプログラミング** とも言える。だが、本当に決定的な手続きなら、自然言語でなく**コード**で書いた方が安い・速い・再現性が高い。

- 「Linear から今日の TODO を取得」→ どの MCP をどの順番で、どの引数で呼ぶかまで明示する
- 「特定の git 操作を 5 回繰り返す」→ シェルスクリプトや Python スクリプトにして、モデルはそれを呼ぶだけにする

これは**オートメーションのピラミッド**の考え方そのもので、再現性と速度が必要な部分はコードに落とし、判断が必要な部分だけを LLM に任せる。スクリプト自体を LLM に書かせてよい。繰り返しの手順をコードにする境界線は、おおよそ「**3 回以上同じことをさせているか**」が目安になる（"Rule of Three" リファクタリング原則の応用）。

### 4. サブエージェントの活用を常に念頭に置く

サブエージェントは、プロンプト／スキル／RULES とは **全く異なる軸** の機能である。前者 3 つが「どう指示するか／どう情報を渡すか」であるのに対し、**サブエージェントはエージェントそのものを分離する**。

#### サブエージェントを使うべき場面

1. **普段メインに持たせたくないツールを使うとき** — MCP を使った外部情報の取得・更新など。
2. **コンテキストを分離したいとき**
   - 独立した調査タスク（最新フレームワークの Web 調査、関連コードの網羅）
   - **結果のみ欲しい**ケース。過程の情報をメインのコンテキストに残したくない。
3. **並列化で高速化したいとき** — ただしこれは品質ではなく速度の話なので、「品質に満足できてから」の最適化として捉える。

Anthropic の "Building Effective Agents" では、このような Subagent の使い分けを **Orchestrator-Workers** および **Evaluator-Optimizer**（評価者 - 最適化者）パターンとして体系化している。特に後者は「メインが計画・評価、Subagent が実行・提案」という形で、コンテキストを分けつつ品質を保てる設計として参考になる。

#### サブエージェントの落とし穴

- **過剰な Subagent 呼び出し** はトークンコストと遅延を押し上げる。Anthropic 自身が「エージェントはトークン消費が大きいので、タスクの複雑さに見合うか検討を」と明記している。
- **Subagent の結果は要約として戻ってくる**。元の証拠（ファイル内容、検索結果の原文）が必要な場合は、メインで直接取得するか、Subagent にソースへのリンクを残させる設計にする。

### 5. プラグインを「配布」だけでなく「集約と切り替え」の道具として使う

Claude Code のプラグインは、外部プラグインを公開したり、組織向けマーケットプレイスを運用したりする文脈で語られがちだ。もちろんそれは重要な用途だが、日常運用ではもっと地味で実用的な使い道がある。**個人用スキル・エージェント・フック群を 1 つの単位にまとめ、必要に応じて有効化・無効化しやすくすること**だ。

公式ドキュメントでも、Claude Code は `.claude/` のスタンドアロン設定とプラグインを並列の仕組みとして案内している。前者は**単一プロジェクト向け・試作向け・個人的な素早い設定**に向き、後者は**複数プロジェクトで再利用したい機能を束ね、名前空間付きで配布・更新したい場合**に向く。したがって「公開するほどではないが、複数リポジトリで同じ自作スキル群を使いたい」というケースでは、プラグイン化はかなり合理的である。

特に便利なのは次の点だ。

- **個人用のスキル群を 1 リポジトリに集約できる**。たとえば Git 運用、レビュー、文章整形、学習支援といった自作スキルを、各プロジェクトに `.claude/skills/` として複製せずにまとめて持てる。
- **有効・無効の切り替えがしやすい**。Claude Code では `/plugin` 画面や `enabledPlugins` 設定で、プラグイン単位に enable / disable を管理できる。つまり「今はレビュー支援プラグインだけ有効」「このリポジトリでは実験用プラグインを切る」といった運用がしやすい。
- **スコープを分けて導入できる**。ユーザー設定で全プロジェクト向けに有効化することも、`.claude/settings.json` や `.claude/settings.local.json` でプロジェクト単位・ローカル単位に寄せることもできる。
- **必要ならマーケットプレイス化して拡張できる**。最初はローカル plugin や個人用 marketplace として始め、後からチーム共有に育てられる。つまりプラグインは「最初から大げさな配布機構」ではなく、**個人設定の整理単位として始めてよい**。

この観点から見ると、プラグインはスキルの代替ではなく、**スキルやエージェントを束ねる配布・切り替えレイヤー**である。レイヤー設計としてはおおむね次のように考えると整理しやすい。

- 単発・試作・その場限り → `.claude/` 直下のスキルや設定
- 複数プロジェクトで繰り返し使う個人資産 → プラグイン
- チームで共有し、導入経路まで含めて標準化したい → プラグイン + マーケットプレイス + 設定ファイル

注意点もある。プラグインはインストールされるとキャッシュにコピーされるため、**プラグイン外のファイルを相対パスで雑に参照する構成は壊れやすい**。また、便利だからといって何でもプラグイン化すると、今度は namespaced skill が増えすぎて発見性が落ちる。したがって実務上は、**まず `.claude/` で試し、再利用したくなったらプラグイン化し、さらに複数人で使う段になったらマーケットプレイスや `extraKnownMarketplaces` / `enabledPlugins` を検討する**、という順番が扱いやすい。

## 発展：複数スキルを整理する一つのアプローチ

スキルを増やしていくと、次の 2 つの運用課題にぶつかる。

1. **スキル一覧の description がコンテキストを圧迫する**。モデル自動起動（`disable-model-invocation: false`）を許したスキルは、全件の description が常時システムプロンプトに載る。スキルが数十個以上に増えてくると、この description 群だけで無視できない負荷になる。
（とはいえ、descriptionの文字数が非常に多くなければ、数十個程度のスキルであればコンテキスト圧迫の観点では問題にならないだろう。それよりも、管理の煩雑さ、AIが目的のスキルを呼ぶ可能性の低下などが真の問題となる）
2. **似たスキルを「まとめて呼び出したい」場面がある**。たとえば「レビュー観点スキル群」を状況に応じて複数同時に参照したい、というニーズ。

このどちらにも使える一つのパターンが、**「グループ用スキル 1 つで、配下のサブスキル群を束ねる」** という設計だ。私の自作スキルであるが、 [`skills/command-group/`](https://github.com/fdshg693/ZENN/blob/main/skills/command-group/SKILL.md) と [`skills/skill-group/`](https://github.com/fdshg693/ZENN/blob/main/skills/skill-group/SKILL.md) が、この考え方の最小実装になっている。

### 設計の骨子

Claude Code のスキル機構は `skills/` 直下しかスキルとして認識しない。つまり **`skills/my-group/sub_skills/foo/SKILL.md` は単独ではスキルとして登録されない**。この制約を逆手に取って、「登録はされないが参照はできる SKILL.md」を `sub_skills/` 配下にぶら下げ、グループ用スキルから一覧・読み込みを仲介する。

この設計の効果は大きい。

- **description を常時ロードから外せる**。サブスキルは Claude Code に認識されないため、description はコンテキストに載らない。モデルが自動で呼ぶこともない。
- **それでも「使いたいときは使える」**。グループ用スキル経由で一覧を取得し、必要なサブスキルだけ SKILL.md を直接 Read して取り込める。

### 2 つの呼び出しモード

同じ「グループ」概念でも、ユースケースに応じて 2 つのモードを使い分けられる。

#### モード A：ユーザーが明示的に複数指定する（`command-group`）

ユーザーが事前に「どのサブスキルを使うか」を知っているケース向け。

```text
/command-group skill1 skill2 skill3
```

のように呼び出すと、スクリプトが指定された各サブスキルの `SKILL.md` を一括で読み込み、コンテキストに投入する。フロントマターで `disable-model-invocation: true` と `user-invocable: true` を指定することで、**モデルが勝手に呼ぶことはなく、ユーザー呼び出し専用**になる。定型プロンプトをまとめて流し込みたいときに相当し、前述の「Claude Code にはプロンプトファイル概念がない」問題の代替にもなる。

#### モード B：AI が探索的に見つけて使う（`skill-group`）

ユーザーが「適切なサブスキルをまだ知らない」ケース向け。

- ステップ 1：グループ用スキルを呼び出すと、サブスキルの **名前一覧** だけが返る（description は含めない）
- ステップ 2：候補に当たりを付けたモデルが `list-skills.sh <name>` で該当スキルの description を取得
- ステップ 3：本当に使うと決めたら `SKILL.md` を直接 Read

このように **「一覧 → 概要 → 内容」** の三段階 Progressive Disclosure を手動で組むことで、全件の description を常時ロードする設計より圧倒的にコンテキストを節約できる。

### この設計の位置づけ

このアプローチは「スキルを増やしすぎたら、スキルを束ねるスキルを作る」という、**設定ファイル自体にメタレベルを一枚追加する**発想だ。ソフトウェア工学でいう「インデックスの導入」「ファサードパターン」に近い。スキルが 10 個を超えたあたりから検討する価値が出てくる。

一点注意すべきは、**サブスキルは Claude Code のスキルとしては呼び出せない**こと。スラッシュコマンド経由や Skill ツール経由の自動起動は効かず、`allowed-tools` などフロントマターの一部指定も無効になる。あくまで「Read 経由で参照される Markdown」として扱う運用に割り切る必要がある。

## まとめ：どこに何を書くか、一枚の判断フローに落とす

最後に、本記事の判断基準を一枚のフローに圧縮する。

1. **プロジェクト全体に常に適用したい規約か？** → RULES レイヤー
   - Claude Code では `CLAUDE.md` 系ファイルと `.claude/rules/`
   - 配置フォルダでスコープしたいならサブディレクトリの `CLAUDE.md`、パスパターンまで切りたいなら `.claude/rules/` の `paths` を使う
2. **ある特定の状況でだけ発動すべき知識か？** → スキル
   - サブファイルに分割できるなら `SKILL.md` は薄いエントリに
   - 手続きが定型化できるならスクリプトに
   - モデルに自動起動させたくないなら `disable-model-invocation: true`
3. **スキルが増えすぎてきた？** → グループ用スキルで束ねる（`command-group` / `skill-group` 方式）、またはプラグインを使った整理を検討する
4. **メインに持たせたくないツール／コンテキストを必要とするか？** → エージェントファイル（Subagent）
5. **長時間タスクで守るべきワークフローか？** → エージェントファイルのシステムプロンプト
6. **日常的に使うエージェントのスタイルを固定したいか？** → エージェントファイル（メインの差し替え）

これらのレイヤーは排他ではなく**積層**で機能する。Claude Code で言えば、`CLAUDE.md` 系ファイルと `.claude/rules/` で土台を作り、スキルで状況別の引き出しを増やし、エージェントでコンテキスト／ツールを分離する。一番上のレイヤーだけで全部をやろうとすると、巨大プロンプトに戻るか、常時コンテキストを浪費するかのどちらかになる。

Anthropic が繰り返し強調する **"do less, better"**（[Effective Context Engineering](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）は、そのままこのレイヤー設計にも当てはまる。**モデルに渡す情報量を最小化するほど、モデルの性能は発揮される**。設定ファイルは「書けば書くほど賢くなる」ものではなく、「適切な層に、適切な粒度で置いたときだけ効く」ものだと捉えるのが、長く付き合うコツだと思う。
