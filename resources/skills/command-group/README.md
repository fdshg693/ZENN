# コマンドグループスキル

## フロントマター

- `disable-model-invocation: true` / `user-invocable: true` により、**ユーザーからの明示呼び出し専用**として動作する
  - AIが自動探索して使うのではなく、ユーザーが必要なサブスキル名を正確に指定して一括投入する用途に寄せているため

## スキルの説明

- `SKILL.md` 内の `!`...`` 記法で `get-skills.sh` を実行し、指定されたサブスキルの `SKILL.md` 内容をそのまま埋め込む
  - `/command-group skill1 skill2 ...` の1回の呼び出しで、複数スキルをまとめてAIのコンテキストへ渡せるようにするため
- サブスキルは `sub_commands/` 配下から動的に探索し、フロントマターの `name` で照合する
  - ディレクトリ名に依存せず、ユーザーが指定するスキル名を識別子の正とするため
- 出力時に各 `SKILL.md` の絶対パスも併記する
  - サブスキル本文中で「同階層の補助ファイルを参照する」ような指示があっても、参照基準を明確にするため
- 存在しないスキル名が含まれていても処理全体は継続し、stderr に警告のみ出す
  - 複数指定時に1件のtypoで全部止まるのを避けるため

## 必要ツール

- Shellコマンドが使える環境
- `bash` / `find` / `sed`
- `yq`（フロントマター解析に使用）

## ディレクトリ構成

```text
command-group/
├── SKILL.md              # スキル本体（ユーザーから /command-group で呼び出す）
├── README.md             # このファイル（スキル設計メモ）
├── PLAN.md               # 設計・実装計画
├── get-skills.sh         # 指定サブスキルの内容を連結出力するスクリプト
└── sub_commands/         # サブスキル配置ディレクトリ
    └── <category-or-name>/SKILL.md
```

## 詳細メモ

### サブスキル取得の仕様（リグレッション防止メモ）

- `get-skills.sh` は `sub_commands/` 配下を **再帰探索**して `SKILL.md` を収集する
  - カテゴリ分けされたサブディレクトリ配下のスキルも扱えるようにするため
- サブスキル名は各 `SKILL.md` フロントマターの `name` フィールドを source of truth とする
  - ユーザーが指定する名前と内部ディレクトリ構成を疎結合に保つため
- 引数は複数受け取り、指定順に `## スキル: <name>` 形式で出力する
  - ユーザーが意図した順序でコンテキストに並べられるようにするため
- 各スキルについて以下を Markdown で出力する
  - スキル名見出し
  - `- path: <SKILL.mdの絶対パス>`
  - `SKILL.md` の全内容（コードブロック囲み）
- 見つからない名前があっても **エラー終了せず警告して継続**する
  - 一部だけ誤っていても、見つかったスキル分はそのまま使えるようにするため

### 呼び出しモデルの制約

- `command-group` は **AIからの自動利用を想定しない**
  - フロントマターで `disable-model-invocation: true` にして、ユーザー主導で必要スキルを明示指定する設計にしている
- このスキルの価値は「探索」ではなく「一括注入」にある
  - 既に必要スキル名が分かっている場面で、複数スキルを1回でAIへ渡すことを優先する
- そのため `skill-group` のような一覧→概要→詳細の段階的探索フローは持たない
  - 柔軟性よりも、手数とコンテキスト投入までの速さを重視するため

### `skill-group` との違い

- `skill-group`
  - AIが利用可能スキルを探索しながら選ぶ用途向け
  - 一覧取得→必要候補の確認→詳細読込、という段階的利用になる
- `command-group`
  - ユーザーが必要スキル名を正確に知っている前提の用途向け
  - 1回の呼び出しで複数サブスキル本文をまとめてAIへ渡せる
