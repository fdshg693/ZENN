## はじめに

### シナリオ:月曜朝の経営会議

ある EC 企業の月曜朝の経営会議で、「先週の新規顧客数」が議題に上がりました。マーケ部門のダッシュボードは **12,400 人**、CS 部門のレポートは **9,850 人**、財務が経理システムから出した数字は **11,200 人**。3 つとも違います。

会議は 40 分かけて「どの数字が正しいのか」を議論し、結局 **次回に持ち越し** になりました。データエンジニアに調査が投げられ、1 週間後に判明したのは「**各部門が独自に『新規顧客』を定義していた**」ことでした。マーケは初回サイト登録、CS は初回問い合わせ、財務は初回決済を基準にしていたのです。

これは特別な事例ではなく、データガバナンスが無い組織で **ごくありふれた症状** です。

#### ガバナンスが「ない」場合に起きること

- **定義のサイロ**: 同じ言葉(顧客、売上、アクティブ)が部門ごとに違う意味を持つ
- **信頼できない数字**: どの数字を信じてよいか分からず、意思決定が遅延する
- **人頼りの情報探索**: 「このデータの出所、誰が知ってる?」が日常
- **過剰権限**: 権限整理がされておらず、誰もが機密データに触れられる
- **監査のたびに大騒ぎ**: リネージや利用履歴を都度手で集める

#### ガバナンスが「ある」場合に変わること

同じシナリオにデータガバナンスが入っていると、次のように変わります。

- **ビジネス用語集**に「新規顧客」の **唯一の定義** があり、各レポートがその定義を参照している
- **データカタログ**で誰でも「新規顧客 KPI」を検索でき、オーナーと品質スコアが並んで見える
- 定義の変更には **データスチュワード** の承認が必要で、履歴が残る
- 数値の食い違いは会議ではなく **事前にダッシュボード側で検出** される
- 監査要求には、**リネージ(どこから来て、どう変換されたか)を数クリックで** 提示できる

会議の 40 分は、数字の答え合わせではなく、施策の議論に使えるようになります。これがデータガバナンスの **実務的な価値** です。

### この記事の目的

「データガバナンス」という言葉はここ数年で一気に一般化しましたが、人によって指すスコープがバラバラです。ある人はアクセス制御を、ある人はデータ品質を、またある人はカタログや GDPR 対応を思い浮かべます。結果として、ツール選定や組織設計の議論が噛み合わないまま進んでしまうことがよくあります。

この記事は、特定の製品やクラウドに寄らない **中立的な観点** で、データガバナンスの

- **定義と目的**(データマネジメントとの違いを含む)
- **なぜ必要か**
- **主要な構成要素(ドメイン)**
- **代表的なフレームワーク(DAMA-DMBOK)**
- **代表的なツール・サービスの立ち位置と連携/運用パターン**

を一枚の地図としてまとめ、議論やツール選定のスタート地点を作ることを狙いにしています。深い実装手順や個別ベンダーの推奨は扱いません。

想定読者は、データ基盤・分析に携わる(または携わり始めた)エンジニア/アナリスト/PMです。

## データガバナンスとは何か

### ひとことで言うと

データガバナンスとは、**「誰が、いつ、どのデータを、どのように扱ってよいのか」を決め、従わせ、検証する仕組み**です。言い換えると「データ版の組織内ルールと統制」に近い概念です。

Tableau の解説は、国家運営の比喩を用いて、データガバナンスを「立法(ルールを決める)・行政(運用を監督する)・司法(遵守をチェックする)」の役割を担うものと説明しています[^1]。NTTコムの記事でも、データガバナンスは「誰がいつどのデータを使用できるかを制御することで、企業全体にわたってビジネスデータの品質とセキュリティを担保する」と定義されています[^2]。

### データマネジメントとの違い

「データマネジメント」と「データガバナンス」は混同されがちですが、役割が異なります。

- **データマネジメント**: データの収集・蓄積・統合・運用・品質向上など、**実際にデータを扱う取り組み全体**。技術的な側面が強い。
- **データガバナンス**: データマネジメント活動に対して、**方針(ポリシー)を決め、評価し、監視する**役割。業務・運用面の統制を担う。

DAMA(Data Management Association International)がまとめた知識体系 **DMBOK** では、データマネジメントの11知識領域のうちの1つとして「データガバナンス」があり、他の領域すべてを調整する **中心的な役割** として位置づけられています[^3][^4]。

> ざっくりまとめ: データマネジメントは「データに何をするか」、データガバナンスは「データに関する意思決定の仕組み」。ガバナンスが欠けたマネジメントは、方針のない現場だけが動いている状態です。

### 主要な役割(ロール)

データガバナンスでは、人の役割分担も重要な要素です。代表的なロールは次のようなものです[^5]。

- **データオーナー(Data Owner)**: あるデータドメイン(例: 顧客、製品)に責任を持ち、アクセスや利用ポリシーの最終判断をする。
- **データスチュワード(Data Steward)**: 日々のメタデータ品質や、ガバナンスポリシーの遵守を運用する。
- **ドメインオーナー / ビジネスオーナー**: 技術的メタデータとビジネス用語をつなぐ、業務と現場の橋渡し役。
- **データガバナンス評議会(Council)**: 部門横断で方針を決めるステアリング組織。

「スチュワードを置かないまま、ツールだけ導入しても形骸化する」というのは、多くの解説が共通して指摘するところです[^6]。

## なぜデータガバナンスが必要か

データガバナンスが欠けていると、データは **信頼されない資産** になります。単に「管理が面倒」という話ではなく、意思決定・規制・セキュリティ・AI活用など複数の方向から必要性が増しています。

### 1. データドリブン経営の土台

ビジネス判断のほとんどがデータに基づくようになった今、データの信頼性が低いと、意思決定はそのまま主観的なものに戻ってしまいます。現場の担当者が「このデータ、信じていいんだっけ?」と毎回自問する状況では、結局「勘と経験」に逆戻りします[^1][^7]。

### 2. 規制遵守(コンプライアンス)

GDPR、各国の個人情報保護法、業界規制(金融・医療・公共など)は、**「どのデータが、どこから来て、どう扱われたか」** を説明できる体制を求めます。監査対応で慌てないためには、リネージ(由来の追跡)やアクセスログが平時から整っている必要があります[^8][^9]。

### 3. セキュリティ / プライバシー

権限が必要以上に広い状態は、内部脅威やランサムウェア時の被害を拡大させます。あるセキュリティ調査では「クライアントの大半が従業員に過剰な権限を付与していた」という指摘もあります[^10]。最小権限・機密度分類・監査可能性といったガバナンスの要素は、そのままセキュリティ強化につながります。

### 4. DX / AI 活用

AI/ML のアウトプットは、学習データの品質・来歴・合意に強く依存します。間違った・古い・偏ったデータを食わせれば、当然モデルも間違えます。モデル出力から訓練データまでの説明責任チェーンを辿れることは、**AI ガバナンスの前提**でもあります[^11]。

### 5. サイロ解消と重複の削減

部門ごとの個別最適で、同じ顧客や製品マスタが **バラバラに定義** されているのは、多くの企業で典型的に起きる症状です。重複レコードはアップセルの機会損失、誤配送、割引交渉力の低下など、具体的なビジネス損害を生みます[^12]。

### ガバナンスがないと現場で何が起きるか

Tableau の記事が列挙する「ガバナンスがない場合の典型症状」は分かりやすいので要約します[^1]:

- 欲しいデータが **どこにあるか分からない**
- データの出所が **不明、信用していいか分からない**
- 形式が **不統一でコンピュータ処理が困難**
- **漏洩が不安でデータを公開できない**

これらは「管理コスト」ではなく、そのまま **意思決定の遅延・誤り・機会損失** に直結します。

## データガバナンスの構成要素

データガバナンスは単一の機能ではなく、複数ドメインの集合体です。以下は、どの解説にも共通して登場するコア要素です。この地図を頭に置くと、自社の課題がどこにあるかが見えやすくなります[^11][^13][^14]。

### 1. ポリシー・役割・組織

- **目的**: 誰がルールを決め、運用し、違反を検知するかを明確にする。
- **主な活動**: データポリシー策定、役割定義(オーナー/スチュワード/評議会)、判断プロセスの明文化。
- **成果物**: データガバナンス憲章、RACI、ガバナンス評議会の議事録。

### 2. データ品質管理

- **目的**: データが「使える」状態を保つ。
- **主な活動**: データ品質の次元(完全性、正確性、一貫性、適時性/鮮度、ユニーク性など)を定義し、測定・監視・改善のサイクルを回す。
- **成果物**: データ品質ルール、スコアカード、SLA。

品質は「やる/やらない」の二値ではなく、**ドメインごとに閾値を定める** ものだと理解するのがコツです。

### 3. メタデータ管理 / ビジネス用語集(Business Glossary)

- **目的**: データの意味・定義・オーナー・利用方法を一元化する。
- **主な活動**: 技術メタデータ(スキーマ、型)とビジネスメタデータ(用語、責任者)の収集・整備。
- **成果物**: 用語集、属性定義、データ辞書。

メタデータは手動更新では破綻します。ETL/ELT、BIレポート、SQL から **自動取り込み(ingestion)** する仕組みが現実的には必須です[^11]。

### 4. データカタログ

- **目的**: 組織内の「データ検索エンジン」として、利用者が必要なデータを見つけ、理解し、信頼できるようにする。
- **主な活動**: データセットの登録、タグ付け、品質スコア・オーナー情報の表示、レビュー/評価。
- **成果物**: 全社データカタログ、アクセス要求ワークフロー。

メタデータ管理が **基盤**、カタログが **ユーザー向けインターフェース** という関係です。

### 5. データリネージ

- **目的**: データが「どこから来て、どこで変換され、どこに流れるか」を可視化する。
- **主な活動**: ソース→変換→格納先の依存関係を自動抽出、影響分析、監査対応。
- **成果物**: リネージグラフ、影響範囲レポート。

リネージは単なる技術的な便利機能ではなく、**規制対応・品質問題の根本原因分析・AIの説明責任** に直接効く領域です。金融・保険のような規制業界では、リネージはコンプライアンス要件そのものになります[^14]。

### 6. マスターデータ管理(MDM) / 参照データ管理

- **目的**: 顧客・製品・サプライヤー・従業員など、ビジネスの中核となる「マスター」を **単一の信頼できるソース** に統合する。
- **主な活動**: 名寄せ(重複統合)、ゴールデンレコードの生成、変更管理。
- **成果物**: マスターデータハブ、参照コード表。

MDM はデータガバナンスと密接ですが、技術的には別物の取り組みとして独立して語られることも多い領域です[^15]。

### 7. アクセス制御・セキュリティ・プライバシー

- **目的**: 機密度に応じた最小権限と、プライバシー保護を両立する。
- **主な活動**: データ分類(公開/社外秘/個人情報/センシティブ等)、ロールベース/属性ベースのアクセス制御、マスキング、監査ログ。
- **成果物**: 分類ポリシー、アクセスマトリクス、監査証跡。

### 8. ライフサイクル管理

- **目的**: データの作成から廃棄までを、規制や保持要件に沿って運用する。
- **主な活動**: 保持期間の定義、アーカイブ、削除(Right to be Forgotten 含む)。
- **成果物**: 保持ポリシー、廃棄記録。

> すべてを同時に始める必要はありません。多くの組織は「まず品質 + カタログ + 用語集」のように、痛みが大きい領域から始めて段階的に広げます。

## 代表的フレームワーク:DAMA-DMBOK

ガバナンスを自前でゼロから設計するのは現実的ではありません。そこで参照されるのが **DAMA-DMBOK** です。

### DAMA と DMBOK

- **DAMA (Data Management Association International)**: 世界各地に支部を持つ非営利団体。
- **DMBOK (Data Management Body of Knowledge)**: DAMA がまとめた、データマネジメントに関する **ベンダーニュートラルな知識体系**。現行は第2版。

### DAMA Wheel と 11 の知識領域

DMBOK の象徴的な図が **DAMA Wheel** です。中央に **データガバナンス** を据え、その周囲に 10 の知識領域が配置されています[^3][^4][^16]。

11の知識領域は以下のとおりです:

1. **Data Governance(データガバナンス)** — 中心
2. Data Architecture(データアーキテクチャ)
3. Data Modeling & Design(データモデリング&設計)
4. Data Storage & Operations(ストレージ&運用)
5. Data Security(データセキュリティ)
6. Data Integration & Interoperability(統合&相互運用)
7. Document & Content Management(ドキュメント&コンテンツ管理)
8. Reference & Master Data(参照&マスターデータ)
9. Data Warehousing & BI(DWH&BI)
10. Metadata Management(メタデータ管理)
11. Data Quality(データ品質)

「ガバナンスが中心」という配置は、**ガバナンスが他領域を所有するのではなく、調整する** ことを意味します[^4]。

### 使い方のコツ

DMBOK は網羅的なので、**そのまま全部を一度に実装するのは現実的ではありません**。Atlan の解説は「正直な成熟度アセスメントから始め、現在のビジネス優先度に直結する領域にフォーカスし、段階的に広げる」ことを推奨しています[^3]。DMBOK は「教科書」というより「地図」として使うのが現実的です。

### その他のフレームワーク(参考)

- **DCAM (Data Management Capability Assessment Model)**: EDM Council が策定。金融機関で採用例が多い。
- **CMMI DMM (Data Management Maturity Model)**: 成熟度モデル系。
- **ISO/IEC 38505 シリーズ**: データガバナンスに関する国際規格(データガバナンスの役割・責任を扱う)。

本記事ではこれ以上深入りしませんが、DMBOK が合わない業界や目的では、上記を検討する価値があります。

## 代表的なツール・サービス概観

ここまでの要素を、実際にどのツールがサポートするのかを見ていきます。どのツールも **データカタログ + ガバナンス統制** を軸にしていますが、得意領域・思想が異なります。

> ⚠️ 注意: 各ツール比較の情報源の多くは、ベンダー系メディア(Atlan、Alation、Collibra、Informatica など自社ブログ)です。当然、自社製品に有利な書き方に寄りがちなので、**同一観点で複数ソースを突き合わせて読む** ことをお勧めします[^17][^18][^19]。

### Collibra

- **立ち位置**: エンタープライズ向け。規制・ガバナンス統制の **深さ** で評価される老舗。
- **強み**: ポリシー管理、ワークフロー、ビジネス用語集、コンプライアンス対応。金融・医療など規制産業での採用が多い。
- **注意点**: モジュール課金 + プロフェッショナルサービス依存で **TCO が高くなりやすい**。インテグレーションでカスタムコネクタが必要な場合あり、との指摘[^19]。

### Alation

- **立ち位置**: 「データカタログ」カテゴリの代表格。ビジネスユーザー向けの **UX と検索性** を強みとする。
- **強み**: 使いやすい検索、導入しやすさ、広範なデータソース連携。
- **注意点**: 深いガバナンス統制やエンドツーエンドのリネージは、Collibra に比べると弱めとの評価[^17]。列レベルリネージは有料・サードパーティ依存になることがある[^17]。

### Informatica(CDGC / Cloud Data Governance and Catalog)

- **立ち位置**: ETL/データ統合の老舗。ガバナンス・カタログ・品質・プライバシーを **統合スイート** として提供。
- **強み**: 大規模統合、ハイブリッド/マルチクラウドでの統合運用、自動データ発見・分類。
- **注意点**: UI が技術寄りで、ビジネスユーザーの採用に時間がかかるとの声[^17][^19]。リネージが Informatica スイート外では弱くなるとの指摘[^17]。

### Microsoft Purview

- **立ち位置**: Microsoft / Azure スタックと一体化したガバナンス基盤。M365 セキュリティ/コンプライアンス製品とも統合。
- **強み**: Azure 上のデータ資産(ストレージ、Synapse、Fabric 等)との親和性、Microsoft ID 基盤や Sensitivity Label との連携。
- **注意点**: **非 Microsoft 環境(AWS/GCP 中心)ではコネクタや機能が限定的** との指摘があり、ビジネスユーザー向け UI も慣れが要るとの評価[^18][^19]。

### Atlan

- **立ち位置**: モダンな「メタデータ・コントロールプレーン」を標榜する新興勢。
- **強み**: 自動化、AI によるメタデータ補助、BI/DWH からカタログ内へのコンテキスト配信、短期導入。
- **注意点**: 深い規制ガバナンス機能、ネイティブコネクタ数は老舗に比べると成長途上で、サードパーティ依存になる部分もある、との指摘[^19]。

### その他よく名前が挙がるもの(簡単に)

- **data.world**: ナレッジグラフベースのデータカタログ。パブリックデータコミュニティも持つ。
- **OvalEdge**: Collibra より安価・導入しやすい中堅向け代替として紹介されることが多い[^18]。
- **BigID**: 機密情報発見・プライバシー分類に強い。ガバナンスというより **データディスカバリ/プライバシー** 寄り。
- **Databricks Unity Catalog / Snowflake Horizon(旧 Governance)**: データプラットフォーム内蔵のガバナンス機能。**自社プラットフォーム内では強力**だが、外部データ資産のカバー範囲は別ツールと組み合わせになることが多い。

ツールカテゴリは流動的で、「カタログ/ガバナンス/品質/MDM/プライバシー」の境界は年々溶けています。単独カテゴリだけで見るよりも、**自組織の構成要素の優先順位(前述の8領域)のどこを埋めたいか** という観点で見るのが実用的です。

## ツールの連携パターンと運用パターン

ツール単体の機能比較以上に重要なのが、**既存のデータスタックとどう連携させるか** と、**組織としてどう運用するか** です。ここを設計せずに製品を入れると、結局は誰にも使われないカタログが出来上がります。

### 典型的な連携パターン

#### パターン1: プラットフォーム内蔵ガバナンス(シングルプラットフォーム型)

- **構成**: Snowflake Horizon、Databricks Unity Catalog のような、**データウェアハウス/レイクハウス内蔵** のガバナンス機能のみで運用。
- **向く場合**: データ資産のほぼすべてが単一プラットフォーム内にあり、BI や他クラウドへの波及が少ないケース。
- **強み**: メタデータ収集もポリシー適用もネイティブ。導入・運用が軽い。
- **弱み**: プラットフォーム外のデータ資産(Tableau、Airflow、別クラウド DWH 等)は視界から外れる[^20]。

#### パターン2: 専任ガバナンスツール中心(横断型)

- **構成**: Collibra / Alation / Informatica / Atlan / Purview などを **中央カタログ** として置き、そこに各データソースのメタデータを集約する。
- **データ取得方法**:
  - **メタデータスキャン**: カタログ側のコネクタで、DWH・BI・ストレージ・ETL のメタデータを定期スキャンして取り込む。
  - **クエリ履歴解析**: Alation のように、**SQL のクエリ履歴を解析して lineage を自動生成** するアプローチ[^21]。
- **向く場合**: データソースが複数クラウド/複数プラットフォームに分散している、ビジネスユーザーが横断的に検索したい場合。

#### パターン3: ハイブリッド(中央カタログ × プラットフォーム内蔵)

- **構成**: 中央に Atlan/Collibra 等、各プラットフォームに Horizon/Unity Catalog 等が並走する形。
- **データフロー**:
  1. 中央カタログがビジネス用語、機密度タグ、オーナー情報を管理
  2. **双方向タグ同期(bidirectional tag sync)** で中央のタグがプラットフォームに伝播
  3. ポリシー(マスキング、アクセス制御)は **プラットフォーム側のエンジン** が実行
  4. リネージは両者を合成してエンドツーエンドで表示
- **具体例**: 中央カタログで "PII-sensitive" タグを付けると、Snowflake Horizon が該当列に自動でマスキングポリシーを適用する、というような連携[^20]。
- **向く場合**: 大規模・複数プラットフォーム運用で、**ポリシーの一元管理と各プラットフォームの高速実行を両立** したい場合。

#### パターン4: プライバシー/分類特化ツール + ガバナンスツール併用

- **構成**: BigID などのプライバシー分類特化ツールで機密情報を発見・分類し、その結果を Collibra/Purview 等のカタログに流し込む。
- **向く場合**: 個人情報保護、GDPR、金融規制など **分類精度が効く領域** が重い場合。

### 典型的な運用パターン

ツールの話と同じくらい大事なのが、「誰がガバナンスを **回す** のか」です。主に 3 つのモデルがあります[^22]。

#### 中央集権(Centralized)

- **形**: 1 つの中央ガバナンスチームがポリシー策定から日々の運用まで担当。
- **メリット**: 一貫性が高い、監査対応しやすい。
- **デメリット**: 中央チームがボトルネックに。現場の業務知識が反映されにくい。
- **向く場合**: 規制産業(金融・医療)、組織が比較的小〜中規模。

#### 分散/連邦(Federated / Decentralized)

- **形**: ドメイン(営業、マーケ、財務など)ごとにオーナー・スチュワードを置く。中央は最小限のルール(メタモデル、共通用語、共通タグ)だけを定義。
- **メリット**: 現場スピードが出る。Data Mesh の発想と相性が良い。
- **デメリット**: ドメインをまたぐ整合性の確保が難しい。
- **向く場合**: 大企業、複数事業部、データ利活用の成熟度が高い組織。

#### ハイブリッド

- **形**: 中央がポリシー・メタモデル・共通用語を所有し、実運用(データの登録・承認・品質監視)はドメインに委譲する。
- **現実解**: 大半の組織はここに落ち着きます。Microsoft Purview の **Domains and Collections** も、この「中央が枠を決め、地域・部門が自分たちの資産を管理する」モデルを前提にした機能設計です[^23]。

### 実装上の典型的な落とし穴

連携パターンと運用パターンを選んだあとに、現場で起きやすいつまずきポイントです。

- **メタデータの手動更新に頼る**: 必ず腐ります。**ETL/SQL/BI からの自動取り込み** を前提にする[^24]。
- **スチュワードを任命だけして権限を渡さない**: 判断権限のないスチュワードは形骸化する。
- **カタログに登録して満足してしまう**: **利用者が検索・発見・信頼判断に使える** 状態で初めて価値が出る。
- **ツールを先に決める**: 痛みの特定 → 構成要素の優先順位 → 運用モデル → ツール、の順で決めないと、高価な「誰も使わないカタログ」になりやすい。

## ツール選定の考え方(短く)

ツール選定は「最強のものを入れる」ではなく、**自組織のガバナンス成熟度と現状の痛み** を起点にスコープを絞るのが定石です。比較記事が共通して挙げる観点は次のあたりです[^18][^19]:

- **規制・業界要件**: GDPR、個人情報保護法、SOX、HIPAA などの強制力と、監査で求められる成果物。
- **既存スタックとの親和性**: DWH(Snowflake/BigQuery/Redshift/Databricks)、BI、ID 基盤、クラウド(AWS/Azure/GCP)。
- **利用者像**: ビジネスユーザー中心か、データエンジニア中心か。UI 要件が大きく変わる。
- **段階導入の可否**: いきなり全社展開ではなく、小さく始められるか。
- **運用負荷**: 専任チームが必要か、既存チームで回せるか。
- **ベンダーロックインと移行容易性**: メタデータ・ポリシーをエクスポートできるか、API は開いているか。

「まず無料/低コスト枠で PoC →痛みが最大のドメイン(例: 顧客マスタ)で勝ちパターンを作る →横展開」という順が現実的です。

## 具体例:Microsoft Purview で見るデータガバナンスの動き方

ここまで中立的に書いてきたので、最後に **具体例として Microsoft Purview を 1 つ取り上げ**、ここまでの構成要素・連携・運用パターンが実際にどうマッピングされるかを見ておきます。

> 本記事では「Purview という製品で何ができるか」を全部網羅するのではなく、**前章までの概念が製品のどの機能に対応するか** だけ追います。

### Microsoft Purview の位置づけ

Microsoft Purview は、**データガバナンス + リスク/コンプライアンス** を横断的にカバーする Microsoft 製品群です。もともと Azure Data Catalog Gen2 + Azure Information Protection を統合する形で誕生した経緯があり、現在は Microsoft 365 / Azure / オンプレ / マルチクラウド(AWS S3、Google Cloud)/ SaaS までを対象にしたガバナンス基盤として位置づけられています[^25][^26]。

### 主要コンポーネントと、本記事の構成要素との対応

Purview のデータガバナンス側の中心は、以下の2つです[^23][^27]。

| 本記事の構成要素 | Microsoft Purview 側のコンポーネント |
|---|---|
| メタデータ管理(基盤) | **Data Map**(全データ資産とメタデータの地図) |
| データカタログ(検索・発見) | **Unified Catalog**(利用者向けの統合カタログ) |
| データ分類 / 機密度 | **Sensitivity Labels + Classification**(スキャン時にサンプリングで分類) |
| リネージ | **Lineage**(ETL/DWH/BI の間で追跡) |
| ポリシー・役割・組織 | **Governance Domains + Collections**(ドメイン単位でポリシーとアクセスを管理) |
| アクセス制御 | **DLP / Protection Policies + Microsoft Entra 連携** |
| プライバシー/規制対応 | **Risk and Compliance ソリューション群** |

### 典型的なシナリオ(イメージ)

例えば「冒頭のシナリオ」のような組織が Microsoft Purview を入れた場合、ざっくり次のような動きになります。

1. **データソース登録とスキャン**
   - Azure Data Lake、Azure SQL、Microsoft Fabric、AWS S3、オンプレ DB などのデータソースを Purview に **登録**。
   - Purview の **Data Map** が定期スキャンし、スキーマ・列・サンプル値から機密情報(メール、クレジットカード番号等)を **自動分類**。
   - リソース量に配慮してスキャンは **業務時間外にスケジュール** することが推奨されています[^27]。

2. **カタログ整備とビジネス用語集**
   - Unified Catalog にビジネス用語(「新規顧客」「売上」など)を登録し、実データ資産に紐付ける。
   - 各資産に **オーナー / スチュワード / 機密度ラベル** を付与。

3. **リネージの自動取得**
   - Azure Data Factory / Synapse / Fabric / Power BI 間のデータの流れを **Lineage として自動取得**[^28]。
   - Power BI ダッシュボードから遡って「この数字はどのテーブルの、どの変換を経て来たか」を追跡できる。

4. **機密度ラベルと保護ポリシー**
   - **Sensitivity Labels** を Microsoft 365(Word/Excel)、Fabric、Data Map で **統一的に運用**[^26][^29]。
   - ラベル付き Fabric アイテムは、エクスポート後も保護が効く。
   - **Protection Policies** でラベルに応じたアクセス制御を自動適用。

5. **ガバナンス運用モデル(Domains + Collections)**
   - **Governance Domains** を部門・地域単位で切り、各ドメインに機密度ラベル、分類、保持、DLP ポリシーを割り当てる[^23][^29]。
   - **Collections** で、その中のデータソースや権限を階層化。
   - 前章の「ハイブリッド運用モデル(中央が枠を決め、各ドメインが自分の資産を管理)」をそのままツール機能として実装した形になっています。

### Purview 採用時に注意したい点

前章で触れた一般論と重なりますが、Purview 特有の注意点として:

- **非 Microsoft 環境での機能成熟度**: AWS/GCP ネイティブのリソースや、SaaS 製品へのコネクタは **時期・対象によって温度差** がある、との評価がある[^18][^19]。多クラウド中心の組織では、対象データソースのサポート状況を **事前確認必須**。
- **スキャンコスト**: Data Map の容量・スキャン時間は対象資産量に比例する。大規模環境では **Collections で対象を分割** し、スキャン頻度を設計する[^23]。
- **Microsoft Entra / M365 / Fabric との統合前提**: Microsoft の他サービスとの統合を前提にすると強みが最大化する。逆に、統合を使わない場合、他ツールに対する相対的な優位性は薄れる。

## まとめ

- データガバナンスは **「ルール・人・プロセス」の総合統制**。技術やツールは後から追従する。
- 「新規顧客の数字が3つ出てくる」ような **日常的に遭遇する痛み** が、ガバナンスの有無で具体的に解消される。
- データマネジメントとの関係は **「実行」vs「統制」**。DAMA-DMBOK では、ガバナンスは 11 知識領域の中心。
- 必要性は **意思決定・規制・セキュリティ・AI・サイロ解消** の5方向から高まっている。
- 構成要素は最低 **8 ドメイン**(ポリシー/品質/メタデータ/カタログ/リネージ/MDM/アクセス制御/ライフサイクル)。全部を同時にやる必要はない。
- フレームワークは **DAMA-DMBOK** をまず地図として使うと整理しやすい。他に DCAM / CMMI DMM / ISO/IEC 38505 も。
- ツールは **得意領域が違う**。選定の前に、**連携パターン(プラットフォーム内蔵 / 中央カタログ / ハイブリッド)** と **運用モデル(中央集権 / 連邦 / ハイブリッド)** を決めておく。
- 具体ツールは比較記事のベンダーバイアスに注意して読む。Microsoft Purview のように、**概念が製品機能にそのまま対応** している例を押さえると理解が速い。
- 選定は「最強」より「自組織の痛み起点」「段階導入可能か」で判断する。

### 次に読むと良いもの

- **DAMA-DMBOK 第2版**: 体系的に学ぶならまず一次ソース。
- **自社が採用するクラウド/プラットフォームのガバナンス公式ドキュメント**: Microsoft Purview、Unity Catalog、Snowflake Horizon など。
- **所属業界の規制**: GDPR、個人情報保護法、業界別ガイドライン。

データガバナンスは「プロジェクト」ではなく「組織能力」です。小さく始めて、使われるものから育てるのが、結局いちばんの近道です。
