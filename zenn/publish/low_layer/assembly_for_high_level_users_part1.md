## この記事について

普段は Python / Go / Java / Rust / JavaScript で仕事をしている。アセンブラなんて、学部の演習以来 10 年触っていない。それでも、こういう質問が来ると 30 秒では答えられない人が多いはずです。

- 「結局アセンブラって機械語と何が違うんですか?」
- 「Go と Rust のバイナリ、同じ x86_64 なのに Windows と Linux で別物なんですか?」
- 「`Release` ビルドで `mov` が消えたって聞いたんですけど、消えていいんですか?」
- 「`cgo` で関数呼ぶときに『呼出規約』とか言われたんですけど、それって何の話ですか?」

この記事は、**アセンブラを書けるようになる記事ではありません**。書けるようにはならないままで結構です。狙いは、

- アセンブラというものが「何の表記法」なのか
- 高級言語が普段、私たちに代わって**何を肩代わりしているのか**
- そして次の後編で扱う「**コンパイラが代行したことの副作用**」を読むための共通言語

を、高級言語ユーザーの語彙でインストールすることです。

前作 2 本で、Python の `.py` から VM のバイトコード、そこからネイティブな C 拡張までは降りました。

- [Pythonは本当にインタプリタ言語か? — `__pycache__`・`.pyc`・バイトコードの裏側を読み解く](./python_bytecode_internals)
- [Python C 拡張の地図 — `PyObject`・参照カウント・GIL から `pybind11`・`PyO3`・abi3・free-threaded build まで](./python_c_extensions_internals)

C 拡張で「ネイティブコード」と書いた最後の一段。本作はそのネイティブコードそのものの正体を扱います。

対象は x86_64 と AArch64。RISC-V や 32-bit 系は触りません。

## 1. 「アセンブラ」と「機械語」と「ISA」をいったん分ける

最初に、よく混ざる 3 つの言葉を分解します。これが混ざっていると先に進めません。

| 用語 | 何か |
|---|---|
| **機械語** | CPU が直接食えるバイト列。`48 89 c8` のような数値 |
| **アセンブラ言語** | その機械語に**ほぼ 1 対 1** で対応する、人間向け表記。`mov rax, rcx` |
| **ISA(命令セットアーキテクチャ)** | CPU が「何という命令を受け付けるか」の契約。x86_64 / AArch64 / RISC-V など |

高級言語が「言語」なら、アセンブラは**言語ではなく「ISA の表記法」**です。Python と Java が違うのと違って、x86_64 のアセンブラと AArch64 のアセンブラは「翻訳系が違う」のではなく、**そもそも対応している CPU が違う**。だから「ポータブルなアセンブラ」は原理的に存在しません。

Compiler Explorer(godbolt.org)を作った Matt Godbolt は、CE の仕組み解説でこう書いています。引用すると、CE は単に「コードをコンパイラに渡してアセンブリ出力を見せる」サービスで、舞台裏ではコンパイラを **4TB** も抱え、各種 ISA / バージョンを並列に走らせている、と[^1]。なぜそんなに種類が要るのか。答えは単純で、**「コンパイラ × ISA × バージョン × 最適化オプション」の組み合わせごとに、出てくるアセンブラは別物だから**です。

[^1]: Matt Godbolt, "How Compiler Explorer Works in 2025", xania.org/202506/how-compiler-explorer-works

つまり、私たちが「アセンブラを読む」と言うときは、暗黙に「**どの ISA の、どのコンパイラの、どの最適化レベルの出力を**読むか」を選んでいます。これが意識から漏れるとずっと話が噛み合いません。

## 2. CPU が見ている世界 — レジスタとメモリと命令ストリーム

高級言語ユーザーは「**変数**」「**関数**」「**ループ**」を扱っているつもりですが、CPU からはそれらは見えていません。CPU の世界には、ざっくり以下しかありません。

- **レジスタ**: CPU 内部の数十個の超高速スロット(x86_64 で汎用 16 個 + α、AArch64 で 31 個 + 0 レジスタ)
- **メモリ**: アドレス付きの巨大な配列。レジスタより 2〜3 桁遅い
- **命令ストリーム**: メモリ上に並んでいる「命令」というバイト列
- **プログラムカウンタ(PC / RIP)**: 次に実行する命令のアドレス

そして、高級言語の概念は次のように CPU レベルへ「翻訳」されます。

| 高級言語 | CPU レベルの実体 |
|---|---|
| 変数代入 | レジスタかメモリへの**書き込み** |
| 変数参照 | レジスタかメモリからの**読み込み** |
| 条件分岐(`if`) | 比較命令 + `PC` を書き換える**条件付きジャンプ** |
| ループ | 末尾のジャンプで `PC` を上に戻す |
| 関数呼び出し | 戻り先 `PC` を保存付きで保存し、関数本体へジャンプ |
| 関数からの戻り | 保存してあった `PC` を復元してジャンプ |

前作で `dis.dis(add)` の出力を見て、Python の `LOAD_FAST` / `BINARY_OP` / `RETURN_VALUE` という抽象スタックマシンの命令を眺めました。CPU の世界はその一段下で、**「スタックマシン」ですらなく、「決まった名前の数十個のレジスタとメモリを使い回す」モデル**だと思ってください。

抽象度の階段で並べるとこうなります。

```
人間の意図:       「a と b を足して返す」
高級言語(Py):     def add(a, b): return a + b
バイトコード(Py): LOAD_FAST a / LOAD_FAST b / BINARY_OP + / RETURN_VALUE
                  ↑ ここまでが前作
                  ↓ ここからが本作
ネイティブ(x64):  mov eax, edi / add eax, esi / ret
機械語:           89 f8 / 01 f0 / c3
```

`a + b` という同じ意図が、抽象度を下げるたびに表現が変わっています。**どの段で見るかで「何が起きているか」の答えが変わる**、というのが大事な発見です。

## 3. `return a + b;` を Compiler Explorer で覗く

論より証拠で、実物を見ます。次の C 関数を考えます。

```c
int add(int a, int b) {
    return a + b;
}
```

これを Compiler Explorer に放り込んで、**x86_64 / Linux の gcc**(`-O2`)で吐かせるとこうなります。

```asm
add:
    lea     eax, [rdi+rsi]
    ret
```

たった 2 命令です。読み方:

- `lea eax, [rdi+rsi]` ... 「`rdi + rsi` の値を `eax` に入れる」(本来 `lea` はアドレス計算用だが、最適化で算術にも使われる)
- `ret` ... 関数から戻る

ここで重要なのは「**`rdi` と `rsi` ってどこから来た?**」です。C 関数の引数として `a` と `b` を渡したはずです。なぜ `rdi` と `rsi` なのか。これは**呼出規約による取り決め**です(後述)。

同じ C コードを、**AArch64**(`-O2`)で吐かせるとこうなります。

```asm
add:
    add     w0, w0, w1
    ret
```

- `add w0, w0, w1` ... 「`w0 + w1` を `w0` に入れる」
- `ret` ... 戻る

レジスタ名が `rdi/rsi/eax` から `w0/w1` に変わり、命令名も変わりました。けれど、抽象度の高い「2 つ受け取って足して返す」という構造は同じです。**同じ意味の C コードでも、ISA が変われば命令もレジスタも別物**——「ポータブルなアセンブラ」が存在し得ない、というのはこういうことです。

ついでに、最適化オプション `-O0`(最適化なし)で吐かせるとどうなるかも見てみましょう(x86_64)。

```asm
add:
    push    rbp
    mov     rbp, rsp
    mov     DWORD PTR [rbp-4], edi
    mov     DWORD PTR [rbp-8], esi
    mov     edx, DWORD PTR [rbp-4]
    mov     eax, DWORD PTR [rbp-8]
    add     eax, edx
    pop     rbp
    ret
```

突然 8 命令に膨らみました。引数をいちいちメモリ(スタック)に書き戻し、また読み出してから足しています。`-O2` ではこの往復を全部省略してくれていたわけです。

ここから得られる、高級言語ユーザー向けの 2 つの事実があります。

1. **同じソースコードでも、最適化レベルで吐かれるアセンブリは大きく違う**
2. **コンパイラは「ソースコード通り」には変換していない**。意味さえ等しければ、メモリ往復もレジスタ転送も自由に消す

この 2 つ目が、後編の「最適化が UB を踏むと爆発する」の伏線になります。

## 4. 高級言語が抽象化した「3 つの仕事」

ここからが本記事の中核です。「コンパイラは何を代行しているのか」を、3 つに分けて言語化します。

### 4.1 レジスタ割付(Register Allocation)

x86_64 の汎用レジスタはだいたい 16 個。AArch64 で 31 個 + α。一方、高級言語ユーザーは平気で 30 個の変数を 1 関数に書きます。

```python
def f(x):
    a = x + 1
    b = a * 2
    c = b - 3
    d = c * c
    e = d + a
    ...
```

「使う変数が増えてもタダ」と感じられるのは、コンパイラが**どの変数をどのレジスタに住まわせるか**を解いてくれているからです。レジスタが足りなくなれば、使用頻度が低い変数を**メモリに退避(spill)**し、必要になったら読み戻します。

レジスタ割付は古典的に NP-hard で、コンパイラ最大の仕事のひとつです。私たちが「変数を増やすかどうか」を性能の観点で気にせずに済んでいるのは、ここを全部丸投げできているからです。

### 4.2 スタックフレームの管理

関数を呼ぶたびに、

- 戻り先アドレス(呼び出し元の `PC`)
- ローカル変数で、レジスタに住み切れなかったもの
- 「使うと決められた」callee-saved レジスタを上書きする前の値

を一時的に置く領域が必要です。これが**スタックフレーム**。各関数が自分用のフレームを「積む / 外す」ことで、再帰や深い呼び出しが成立しています。

x86_64 では `%rsp` レジスタが「スタックの先端」を指し、**スタックは下方向に伸びます**[^2]。`push` でアドレスが小さい方へ動き、`pop` で大きい方へ戻る。Harvard の CS 61 講義の表現を借りると:

[^2]: cs61.seas.harvard.edu/site/2018/Asm2/ "x86-64 stacks look like this... a 'push' operation adds space ... by moving the stack pointer left, to a numerically-smaller address"

> x86-64 では、各スタックフレームは **16 バイトの倍数**にアラインされる必要があり、`callq` 命令の実行開始時、`%rsp` は **16 バイトアラインされていなければならない**。つまり関数エントリ時点の `%rsp` は「16 の倍数から 8 バイトずれた値」になる。

これだけ読んで「面倒くさ」と思った方、正しいです。**この面倒くささを 100% 代行している**のがコンパイラの 2 つ目の仕事です。

高級言語ユーザーは、

```go
func outer() int {
    return inner(1, 2) + 3
}
```

と書くだけで、`outer` と `inner` の両方で必要なフレーム確保・解放・アラインを意識せずに済みます。これが**ゼロでない仕事**だ、という認識だけ持ち帰ってください。

### 4.3 呼出規約(Calling Convention)/ ABI

3 つ目が、いちばん表に出にくいけれど一番ややこしい代行業です。

関数を呼ぶ側(caller)と呼ばれる側(callee)は、**事前に**以下を合意しておく必要があります。

- 引数をどのレジスタやスタック位置に置くか
- 戻り値はどこに置くか
- 呼び出しによって**自由に壊してよいレジスタ**(caller-saved / volatile)
- 呼び出しを跨いで**保存すべきレジスタ**(callee-saved / nonvolatile)
- スタックのアライメント要件、戻り時のスタック状態
- 構造体や大きすぎる値の渡し方
- 浮動小数点や SIMD 用レジスタの扱い

この合意のことを**呼出規約**、より広い「同じバイナリが連携できるための取り決め」を **ABI**(Application Binary Interface)と呼びます。

ここで衝撃的なのは、**同じ x86_64 でも、OS が違えば呼出規約は別物**だという事実です。Linux/macOS は **System V AMD64**、Windows は **Microsoft x64**。`gcc -O2` で `add(a, b)` を吐かせると Linux では引数が `rdi/rsi`、Windows では `rcx/rdx` になる、というのはここから来ます。

そして AArch64 にはまた別の規約、**AAPCS64**(Procedure Call Standard for the Arm 64-bit Architecture)があります。Arm 公式は PCS について端的にこう書いています[^3]。

[^3]: developer.arm.com/documentation/102374/0103/Procedure-Call-Standard "Arm architecture places few restrictions on how general purpose registers are used... if you want your code to interact with code that is written by someone else, or with code that is produced by a compiler, then you need to agree rules for register usage. For the Arm architecture, these rules are called the Procedure Call Standard, or PCS."

> Arm アーキテクチャは汎用レジスタの使い方にほとんど制約を課さない。しかしあなたのコードが「他人が書いたコード」や「コンパイラが生成したコード」と相互作用するなら、レジスタ使用ルールに合意する必要がある。Arm ではこれを **Procedure Call Standard, PCS** と呼ぶ。

つまり、呼出規約とは「**他人と一緒に動くために、コンパイラ全社で守っている公式契約**」です。

## 5. 呼出規約 3 種を並べてみる

詳細は本記事の主旨ではないので、最低限の対比だけ表で示します。

| 項目 | **System V AMD64**<br/>(Linux/macOS) | **Microsoft x64**<br/>(Windows) | **AAPCS64**<br/>(AArch64) |
|---|---|---|---|
| 整数引数レジスタ | `rdi, rsi, rdx, rcx, r8, r9`<br/>(最大 6) | `rcx, rdx, r8, r9`<br/>(最大 4) | `x0`〜`x7`<br/>(最大 8) |
| 戻り値 | `rax`(+ `rdx`) | `rax` | `x0`(+ `x1`) |
| caller-saved の例 | `rax, rcx, rdx, rsi, rdi, r8-r11` | `rax, rcx, rdx, r8-r11` | `x0-x15` |
| callee-saved の例 | `rbx, rbp, r12-r15` | `rbx, rbp, rsi, rdi, r12-r15` | `x19-x28` |
| スタック特殊事情 | 関数末尾の **128 バイト red zone**(関数中で `rsp` を動かさず使ってよい領域) | 呼び出し元が **32 バイトの shadow space** を `call` 前に確保 | `sp` は **16 バイトアライン**必須 |

ここで注目してほしいポイントを 2 つ。

**ポイント 1: Linux と Windows で、`rdi` と `rcx` のどっちが第 1 引数かが違う**

Microsoft x64 の仕様には次のような記述があります[^4]。

[^4]: learn.microsoft.com/en-us/cpp/build/x64-calling-convention "uses a four-register, fast-call calling convention by default. Space is allocated on the call stack as a shadow store for callees to save those registers."

> x64 はデフォルトで **4 レジスタの fast-call** 呼出規約を使う。呼び出し元のスタックには callee がレジスタ退避できる **shadow store** を確保する。

System V AMD64 は 6 レジスタ。Microsoft x64 は 4 レジスタ + shadow space。**同じ CPU、同じ命令セット、同じバイナリ形式に近い世界なのに、関数を呼ぶ作法が違う**。Wikipedia の整理を借りるなら、Microsoft x64 では `RAX, RCX, RDX, R8, R9, R10, R11` が caller-saved(volatile)、`RBX, RBP, RDI, RSI, RSP, R12–R15` が callee-saved(nonvolatile)です[^5]。System V では `RDI, RSI` は callee-saved ではなく**引数渡し用の caller-saved**。同じレジスタ名でも、OS が違うだけで役割が逆になっている。

[^5]: en.wikipedia.org/wiki/X86_calling_conventions

**ポイント 2: 「8 バイトに収まらない引数」は特殊扱いされる**

Microsoft x64 公式にはこうあります[^4]。

> 引数は 1, 2, 4, 8 バイト以外、あるいは 8 バイトに収まらないものは**参照渡し**になる。単一の引数を複数レジスタに分割して渡すことは無い。

つまり、64-bit に収まらない構造体は「**呼出元が確保したメモリへのポインタ**」として渡る。戻り値も同様で、大きな構造体を返すと、見た目は `return s;` でも実態は「呼出元があらかじめ確保したバッファアドレスを `rcx` に渡して、callee がそこに書き込む」という変換が起きる。

この「**ソースコードの値渡しが、ABI 層では参照渡しに化ける**」現象は、後編の FFI バグの主要因になります。

## 6. まとめ — 高級言語は「CPU との対話」を抽象化した言語である

ここまでで、コンパイラが私たちに代わって担当している 3 つの仕事を見ました。

1. **レジスタ割付**: 数十個しかないレジスタへの変数の住み分けと spill
2. **スタックフレーム管理**: 戻り先・ローカル・退避レジスタを置く 16 バイトアラインの領域確保
3. **呼出規約 / ABI の遵守**: OS と ISA ごとに違う「他人と協調するための作法」

高級言語が解決した問題は、ざっくり以下です。

- **ポータビリティ**: ISA や OS が違っても同じソースで通る(コンパイラがそれぞれの ABI を吐いてくれる)
- **認知負荷**: レジスタ・スタック・呼出規約を意識しなくていい(自由変数として書ける)
- **配布性**: Windows / Linux / macOS / iOS / Android で「コンパイルし直すだけ」で動く可能性が出る

でも、ここまで読んできて気付いた人もいるはずです。**これは無料ではない**。コンパイラが介在することで、ソースコードに書いた「意味」と、実際に CPU 上で走るネイティブコードの「意味」のあいだに**ズレ**が生まれます。

- `-O0` と `-O2` で命令数が 4 倍違う
- 同じ C コードが Linux と Windows で別レジスタ・別フレーム構造になる
- 8 バイト超の構造体は黙ってポインタ渡しに化ける
- メモリ往復は最適化で消える(あるいは復活する)

このズレを放置すると、現代の典型的なバグや性能事故を踏みます。

- **「アルゴリズム的には同じはず」なのに 5 倍遅い**
- **`-O2` を付けたら壊れる**
- **マルチスレッドで「あり得ない順序」が観測される**
- **Rust から C を呼ぶと segfault する**

これらは、ソースコードを読むだけでは絶対に見えません。コンパイラが代行した「翻訳」の中で起きているからです。

後編では、この「**抽象化の漏れ(Leaky Abstraction)**」を 5 ケースに分けて整理します。

- キャッシュライン
- 分岐予測 / 投機実行
- undefined behavior と最適化
- メモリモデル(並行性)
- ABI / FFI バグ

各ケースで「高級言語ユーザーがいつ降りる必要があり、いつ降りなくていいか」も合わせて扱います。
