## この記事について

前編 [アセンブラを書かない人のためのアセンブラ — 高級言語が抽象化した 3 つの問題](./assembly_for_high_level_users_part1) で、コンパイラが私たちの代わりに代行している仕事を 3 つ整理しました。

1. レジスタ割付
2. スタックフレーム管理
3. 呼出規約 / ABI の遵守

「代行されている」ということは、ソースコードと、実際に CPU 上で走るネイティブコードのあいだに**翻訳の層**があるということです。普段はそれで何の問題もありません。問題が起きるのは、その翻訳のどこかから**抽象化が漏れる**瞬間です。

この後編で扱うのは、高級言語ユーザーが現場で踏みうる「漏れ」の 5 ケースです。

- キャッシュライン
- 分岐予測 / 投機実行
- undefined behavior と最適化
- メモリモデル(並行性)
- ABI / FFI バグ

最終的に、「**いつ低レイヤーに降りるべきか / いつ降りなくていいか**」までを言語化することを目標にします。

## 1. Leaky Abstraction を共通言語にする

2002 年に Joel Spolsky が書いた古典に "The Law of Leaky Abstractions" があります。要約すると、

> どんな自明でない抽象も、ある程度まで**漏れる**。

これは批判ではなく**観察**です。抽象を完璧にしようとすると、たいてい「抽象の外側」の細部を全部抽象の中に持ち込む必要が出てきて、結局は抽象として機能しなくなる。だからプラクティカルな抽象は必ずどこかから漏れます。

高級言語の場合、漏れるのはだいたい次の 5 箇所です。

| 漏れ方 | どこから漏れる | 高級言語ユーザーが踏む典型症状 |
|---|---|---|
| 性能 ① | キャッシュライン | 同じ計算量なのに数倍遅い |
| 性能 ② | 分岐予測 / 投機実行 | データ順序や `if` 配置で速度が変わる |
| 意味論 ① | undefined behavior | `-O0` で動くのに `-O2` で壊れる |
| 意味論 ② | メモリモデル | マルチスレッドで「あり得ない順序」が見える |
| 境界 | ABI / FFI | 言語境界で segfault / 値が化ける |

順に見ていきます。

## 2. 性能の漏れ ① — キャッシュライン

CPU はメモリを 1 バイトずつ読みません。**まとめて 64 バイト**読みます。この 64 バイトの塊が**キャッシュライン**で、現代のほとんどの CPU でこのサイズです。

これが効いてくる典型例:

```c
// パターン A: 配列を順に舐める
for (int i = 0; i < N; i++) sum += arr[i];

// パターン B: ランダムな順序で同じ要素にアクセスする
for (int i = 0; i < N; i++) sum += arr[perm[i]];
```

両方とも `O(n)` です。アクセス回数は同じ。にもかかわらず、`N` が十分大きいと B は A の 5〜10 倍遅くなります。理由は、A では「最初の 1 バイトを読みに行ったついでに 64 バイトキャッシュに載った」結果、次の 15 回(`int` なら)の読み出しがキャッシュヒットになる。B はジャンプするたびに L1 ミスする。

これは「アルゴリズム的には同じ」のに速度が桁違いになる**性能の漏れ**です。アセンブラを読んでも `mov eax, [rdi+rsi*4]` のように同じ命令が並ぶだけで、「**なぜ**この命令が遅いか」はアセンブラの上には書いていません。CPU のキャッシュ階層という別の層を知らないと見えない。

### 配列の中身が「構造体の配列」なら、もう一段の罠がある

```c
struct Particle { float x, y, z; int id; char tag[48]; };  // 64 byte
Particle particles[N];

// x だけ全部足したい
for (int i = 0; i < N; i++) sum += particles[i].x;
```

ループは `x` しか読まないのに、CPU はキャッシュラインごと持ってくるので、毎回**使わない `tag[48]` まで運んでいる**。ここで Struct of Arrays(`xs[N], ys[N], zs[N], ...`)に組み直すと、`xs` だけが密にキャッシュに乗って速くなる。ゲームエンジンや数値計算でよく出る話です。

### マルチスレッドだと、もっと意地悪な漏れがある — False Sharing

```c
struct Counters {
    long a;  // スレッド A が更新
    long b;  // スレッド B が更新
};
```

`a` と `b` は論理的には独立で、どちらのスレッドも相手のフィールドには触りません。だから**並列に動くはず**です。

実際に走らせると、シングルスレッド実行より遅くなることがあります。これが **False Sharing** です。`a` と `b` が同じ 64 バイトキャッシュラインに乗っていると、

1. スレッド A がコア 1 で `a` を書き込む
2. その瞬間、コア 2 の同じキャッシュラインは**全部無効化**される(`b` も巻き添え)
3. スレッド B が `b` を読みに行くと L1 ミス → メモリ階層を経由して取り直し
4. ステップ 1 に戻る

各スレッドが、知らないところでお互いのキャッシュを叩き落とし続ける。これを業界では **cache line ping-pong** と呼びます[^1]。

[^1]: aussieai.com/blog/false-sharing "If one thread sets any value in a 64-byte cache line, then all of the other 63 bytes are also invalidated in the cache. If a second thread needs to use any of those other 63 bytes, then it needs a cache line refresh."

解決策は単純で、フィールドの間に 64 バイトのパディングを入れるか、C++ なら `alignas(64)` を使う。

```cpp
struct Counters {
    alignas(64) long a;
    alignas(64) long b;
};
```

この一言で性能が一桁戻ることがあります[^2]。

[^2]: dev.to/ariasdiniz/understanding-and-solving-false-sharing-in-multi-threaded-applications-with-an-actual-issue-i-had-57c

「キャッシュライン」という抽象は、ソースコードには出てきません。`struct Counters` を見ても 64 バイト境界の話は書いていない。アセンブラを見ても `mov` 命令が並んでいるだけで、それがキャッシュヒットなのかミスなのかは命令の上では分からない。これが**性能の漏れ**です。

### 高級言語ユーザーへの含意

- **データ局所性は計算量と独立に効く**。`HashMap<K, V>` を `Vec<(K, V)>` に変えただけで速くなる現象の正体はここ
- **マルチスレッドで同じ構造体に複数スレッドが書く設計は危険**(False Sharing が起きうる)
- 数値処理が遅いとき、AoS(Array of Structs)を SoA(Struct of Arrays)に組み直す価値がある

## 3. 性能の漏れ ② — 分岐予測 / 投機実行

現代の CPU は、命令を上から順に 1 つずつ実行していません。**パイプライン**化されていて、フェッチ・デコード・実行・書き戻しが重なって進む。さらに**結果が出る前に**次の命令を投機的に進めます。

そこで問題になるのが**条件分岐**です。`if` に出会うと「どちらに飛ぶか」は、比較が終わるまで本当は分からない。けれど CPU はそこで止まりません。「**たぶんこっちに飛ぶだろう**」と予測して、その先を実行し始める。これが **branch prediction** と **speculative execution** です[^3]。

[^3]: en.wikipedia.org/wiki/Branch_predictor

予測が当たれば速い。外せば、

- 投機実行で進めた数十命令分の結果を**全部破棄**する
- パイプラインをフラッシュして、正しい分岐先からやり直す

このペナルティが**十数〜数十サイクル**。1 ns の世界では大きな代償です。

### ソート済み配列のほうが速い理由

StackOverflow で有名な問いがあります。「同じ計算なのに、ソート済み配列を処理するほうが速いのはなぜか」。

```c
for (int i = 0; i < N; i++) {
    if (arr[i] >= 128) sum += arr[i];
}
```

`arr` がソート済みなら、ある時点までは `if` が常に偽、それ以降は常に真。**分岐予測は 99% 当たる**。ランダムだと毎回外れる(50%)。結果として 5〜6 倍速度差が出る、というやつです。

### 「常に分岐予測が外れる = 即遅い」とは限らない

ここはニュアンスが要ります。Johnny's Software Lab の実験データに次のような観察があります[^4]。

[^4]: johnnysswlab.com/how-branches-influence-the-performance-of-your-code-and-what-can-you-do-about-it/ "The branch in our binary search cannot be predicted well, yet when there is no data prefetching our regular algorithm performs the best. Why? Because branch prediction, speculative execution and out of order execution give the CPU something to do while waiting for data to arrive from the memory."

> バイナリサーチの分岐はうまく予測できない。にもかかわらず、データプリフェッチが効いていないとき、普通のバイナリサーチ実装が一番速い。理由は: 分岐予測 + 投機実行 + アウトオブオーダー実行が、**メモリ待ちのあいだに CPU に何かやらせる**装置になっているから。

つまり、分岐予測は単に「予測の正解率」を稼ぐ装置ではなくて、「**メモリ律速のときに、待ち時間を投機実行で隠す**」働きをしている。だから「branchless にすれば常に速い」と短絡してはいけない。

### 高級言語ユーザーへの含意

- **データを事前にソートすると劇的に速くなる**ケースが現実にある
- **ホットパスから `if` を消す**(=branchless にする)テクニックには意味があるが、メモリ律速のときは逆効果
- JIT / AOT がやっている **PGO(Profile-Guided Optimization)** の動機はここ。本番の分岐傾向を計測してから最適化するほうがよく当たる
- 性能チューニングで「アルゴリズム計算量は同じはずなのに速度が違う」を見たら、最初に疑うべきはキャッシュと分岐予測

ここまでが「性能の漏れ」です。次は「**意味論の漏れ**」、つまり「コードに書いた意味そのものが守られないケース」に進みます。

## 4. 意味論の漏れ ① — undefined behavior

ここからが、高級言語ユーザーがいちばん油断する領域です。

C / C++ には **undefined behavior (UB)** という規約上の概念があります。よく誤解されるのですが、UB は

- ❌ 「クラッシュする」
- ❌ 「未実装」
- ❌ 「実装依存(implementation-defined)」

のいずれでもありません。**「言語仕様が何の制約も課さない」**という、コンパイラに渡された白紙小切手です。コンパイラは「UB が起きるコードは**書かれない**」と仮定して最適化してよい、と仕様に書かれています。

「ちょっと壊れるくらい平気でしょ」と思った人へ。次の例を見てください。

### Russ Cox の `rm -rf slash` 事件

これは Russ Cox がメモリモデルの解説の中で挙げた、有名な実例の構造です[^5]。

[^5]: research.swtch.com/plmm

```c
typedef int (*Fn)();
static Fn Do;

int EraseAll() {
    return system("rm -rf slash");
}

void NeverCalled() {
    Do = EraseAll;
}

int main() {
    return Do();
}
```

`Do` は最初 `NULL`。`NeverCalled` は名前の通り**呼ばれない**。普通に読めば `main` は **`NULL` を呼んで segfault する**プログラムに見えます。

ところが Clang はこう最適化します。

1. `main` で `Do()` を呼ぶ。`Do` は `NULL` か `EraseAll` のどちらか
2. もし `NULL` なら、関数ポインタ経由の null 呼び出しは **UB**
3. UB は「起きないと仮定してよい」。だから `Do` は `EraseAll` であるはず
4. ならば `Do()` は `EraseAll()` の直接呼び出しに置き換えてよい
5. ついでに `EraseAll` をインライン化

結果として吐かれるのは、こう。

```c
int main() { return system("rm -rf slash"); }
```

**呼ばれてもいない関数の中身が、main の本体に化けた**。ソースコードを上から下まで読んでも、`main` から `system` が呼ばれる経路はどこにもありません。それでもバイナリは `rm -rf` を確実に実行します。これが UB が「白紙小切手」だということの意味です。

これは作為的な例ではあります。ただし、John Regehr が書いているとおり、こういう構造の現実の発生例はいくらでも報告されています[^6]。「`if (p && p->next)` の `p &&` を最適化が消す(`p->next` を踏める時点で `p` は非 null だと推論される)」「整数オーバーフローを `if` で防ごうとしたコードが、オーバーフローは UB なので `if` ごと最適化で消される」などが典型です。

[^6]: blog.regehr.org/archives/1520 "Undefined Behavior in 2017"

### 「`-O0` では動いていたのに `-O2` で壊れる」の正体

UB を踏んでいるコードは、最適化レベルが弱いと「**たまたま動く**」ことがあります。コンパイラがそこまで強い推論をしないからです。最適化を上げると、コンパイラは UB を「絶対に起きない仮定」として強く使う。結果として、最適化を上げた途端コードが**意味的に変質**する。

> "It is much easier to say that unexpected changes to 'q' will cause UB, than to specify the range of consequences that may occur."[^6]

UB の怖さは「**結果が予測できない**」ではなく、「**コンパイラの推論連鎖がどこまで広がるか、誰にも見積もれない**」ことにあります。

### 高級言語ユーザーへの含意

- C / C++ や Rust の `unsafe` ブロックを書く瞬間、**ソースコードの素直な意味論はもう信用できない**
- 「動いているように見えるから OK」は最も危険な反応。UB は別のコンパイラバージョンや別の最適化フラグで突然顔を出す
- Rust の safe な範囲、JVM、CPython、Go のような managed 言語は、この爆弾を「**踏ませない**」ことに価値がある。Rust の `unsafe` が**強く目立つ構文**になっているのも、ここを境界化するため
- C/C++ プロジェクトでは sanitizer(UBSan, ASan, TSan)を CI で常時走らせる価値がある。「テストが通る」は UB に対しては保証にならない

## 5. 意味論の漏れ ② — メモリモデル(並行性)

シングルスレッドの世界では、コンパイラは「**観察可能な意味を変えない限り、命令を自由に並べ替えてよい**」というルールで動いていました。前編で `-O0` と `-O2` の出力命令数が違う、という話の正体はこれです。

ところがマルチスレッドの世界では、「観察可能」の定義そのものが揺らぎます。

### 並べ替えは「3 段」起きる

並列スレッドが見る順序は、書き手のソースコードの順序とは限りません。途中で**並べ替え**が 3 回起きます。

1. **コンパイラの最適化**: 命令の順序を変える(レジスタ再利用、共通部分式の括り出しなど)
2. **CPU の out-of-order 実行**: 依存関係がない命令を、書かれた順序と関係なく実行する
3. **キャッシュ間の伝播**: コア A の書き込みがコア B から見えるまでの遅延、見える順序の揺らぎ

シングルスレッドなら、各段は「最終結果がソースコード通り見える」ように調整されます。マルチスレッドだと、別スレッドから見たときの観測順序は**保証されない**。

### 古典的な Store Buffering の例

初期値が両方 0 で、別スレッドで以下を同時実行する場合:

```text
Thread A:           Thread B:
x = 1;              y = 1;
r1 = y;             r2 = x;
```

直感的には「**`r1 == 0 && r2 == 0` はあり得ない**」と感じます。少なくともどちらかのスレッドは相手の書き込みを見るはず、と。

実機では `r1 == r2 == 0` が観測できます。Russ Cox はこう書いています[^7]。

[^7]: research.swtch.com/plmm

> sequentially consistent atomics と acquire/release atomics の違いは、後者では「両方ゼロ」のような観測が許される点にある。

理由は、各 CPU が「自分の書き込みを **store buffer** に一旦置く」「他コアから見えるまでにラグがある」「Thread A の `x = 1` を Thread B が見るまでに、Thread A は `y` の値を**古いまま**読んでしまう」ような順序が物理的に成立しうるからです。

### `acquire / release` というのは何の話か

現代の言語(C++11, Java, Go, Rust)は、「**全部の書き込みが順序通り見える(sequential consistency)**」を諦め、書き手が**必要なバリアだけ宣言する**モデルを採用しています。

- `relaxed`: 順序保証なし(普通の読み書きとほぼ同じ)
- `acquire`: この読み込み以降の読み書きは、これより上に移動しない(=ロック取得時の意味)
- `release`: この書き込み以前の読み書きは、これより下に移動しない(=ロック解放時の意味)
- `seq_cst`: すべてのスレッドから見て一貫した順序

通常の Mutex や RwLock は、内部でこの `acquire/release` バリアを発行しています。「Mutex は値を壊さない装置」ではなく、「**値を壊さないために必要なバリアを発行する装置**」だと理解するのが正しい。

### 高級言語ユーザーへの含意

- **「ロックなしで `flag = true` してから値を書いた」は危ない**。値の書き込みが先に他スレッドに見える順序が物理的に起こりうる
- `Mutex` / `Lock` / `synchronized` を使うのは、データ競合を「壊れない」にするだけでなく、**並び替えを抑える**ためでもある
- Python の GIL は、この問題の多くを「踏ませない」装置でもあった(前作 [C 拡張の地図](./python_c_extensions_internals) で触れた話と接続する)。3.13 以降の free-threaded build で C 拡張側が壊れる典型理由のひとつがこれ
- データ競合のテストは「テストが通る」では保証にならない。`r1 == r2 == 0` は**走らせるたびに観測されるとは限らない**。CI で 10 万回回しても見えず、本番で初めて顔を出すことがある

ここはアセンブラを読んでも見えません。アセンブラは「命令の順序」を示すだけで、「**他コアから見える順序**」を示せないからです。メモリモデルという**CPU よりさらに上の規約**の話になります。

## 6. 境界の漏れ — ABI / FFI バグ

最後は、前編で触れた呼出規約が**自分の方に降りかかってくる**ケースです。FFI、つまり言語境界です。

### 典型ケース 1: 構造体パディングの食い違い

```c
// C 側
struct Point { char tag; int x; int y; };
```

C コンパイラは `int x` を 4 バイト境界に置きたいので、`tag` の後ろに**暗黙の 3 バイトパディング**を入れます。`sizeof(struct Point)` は 12 になる(1 + 3 + 4 + 4)。

Python の `ctypes` や Rust の `bindgen` は通常これを正しく扱いますが、**手で `repr(C)` を書いて宣言する場合**にミスると、x のオフセットを 1 と仮定してしまい、読み出した値はゴミです。Segfault ではなく**静かに値が化ける**ので発見が遅れます。

### 典型ケース 2: 「`int` のサイズ」が言語間で揺れる

C の `long` は、ILP64 / LP64 / LLP64 の選択で別物です。

| OS / プラットフォーム | `long` のビット幅 | データモデル |
|---|---|---|
| Linux x86_64 | 64 ビット | LP64 |
| macOS x86_64 / AArch64 | 64 ビット | LP64 |
| Windows x86_64 | **32 ビット** | LLP64 |

「同じ x86_64」「同じ C コード」でも、Windows と Linux で `long` のサイズが違う。Rust から `extern "C" { fn foo(x: c_long) }` を呼ぶときに、対応する C 側関数の `long` が両方の OS で扱えるサイズを取っていないと、片方で値が切り詰められます。

### 典型ケース 3: 構造体の戻り値が黙ってポインタ渡しに化ける

前編で触れた Microsoft x64 仕様より:

> 引数や戻り値は 1, 2, 4, 8 バイト以外、あるいは 8 バイトに収まらない非 POD は**参照渡し**になる。

「16 バイトの struct を値で返す」C 関数を、Python `ctypes` で**戻り値型を `c_int` と誤宣言**して呼ぶと、

- 呼出側は「戻り値を `rax` で受け取る」つもり
- 呼ばれた側は「呼出元が確保したメモリへのポインタが第 1 引数で渡ってくる」前提で書き込む
- レジスタと引数が**全部 1 つずつズレる**

このとき発生するのは、典型的には**呼ばれた関数が壊した callee-saved レジスタの巻き戻しに失敗 → スタック破壊 → 戻った後に segfault** という、原因とクラッシュ箇所が無関係な事故です。

### 典型ケース 4: 呼出規約の取り違え

Windows の Win32 API は `__stdcall`、C ランタイムは `__cdecl`、というレガシーな違いがあります。x86(32-bit)時代は呼出元と呼ばれた側のどちらがスタックを掃除するかが規約で異なり、間違えるとスタックポインタが**ずれたままスレッドが進む**バグになりました。

x86_64 ではこれは概ね統一されましたが、COM や一部の Windows API では今でも特定の規約を要求します。Rust の `extern "system"` や Go の `syscall` パッケージが微妙に違うのはこのため。

### 高級言語ユーザーへの含意

- 「Python から C 拡張」「Rust から C / C++」「Go から `cgo`」「Java から JNI」を書く瞬間、**前編で見た ABI を自分でやる側に立たされている**
- ライブラリの自動生成(`bindgen`, `cffi`, `pybind11`, `cgo` etc.)に頼れる部分は最大限頼る
- バインディングを手書きする場合、**`sizeof` と `offsetof` を C 側で出力して付き合わせる**のが最も信頼できる検証法
- segfault がランダムに見えるとき、まず**ABI 不一致(構造体レイアウト、長さ、呼出規約)**を疑う

## 7. まとめ — いつ降りるべきか / いつ降りなくていいか

ここまで読んだ後で、「では明日からアセンブラを学ぶか」というと、ほとんどの場合**学ばなくていい**です。むしろ降りなくていい場面のほうが圧倒的に多い。

### 降りなくていい場面

- **アプリケーションロジック、CRUD、ビジネスルール**: ここで効くのは設計と可読性。アセンブラは関係ない
- **I/O bound のコード**: DB / ネットワーク / ディスクが律速のとき、CPU 側の最適化は誤差
- **アルゴリズム計算量が悪い問題**: `O(n²)` を `O(n log n)` にする方が、低レイヤー最適化より遥かに効く
- **テストで再現できるバグ**: ロジックバグはソースコードの上で潰せる

### 降りる価値がある場面

- **アルゴリズム計算量は最適なのに性能が出ない**: キャッシュ / 分岐予測 / SIMD を疑う
- **マルチスレッド / lock-free の正しさが要る**: メモリモデルの理解が必須
- **C / C++ / Rust unsafe / FFI を書く瞬間**: UB と ABI を意識せざるを得ない
- **`Release` ビルドだけ挙動が変わる、`-O0` と `-O2` で違う**: UB の可能性が高い
- **言語境界で segfault する / 値が化ける**: ABI 不一致を疑う

### 抽象度の階段、最終版

前作 2 本 + 前編 + 本作の 4 部作で、私たちは抽象度の階段を降りてきました。

```
人間の意図
   ↓
ソース(.py / .c / .rs / .go)
   ↓                        [前作①: Python バイトコード編]
バイトコード / IR
   ↓                        [前作②: C 拡張編]
ネイティブコード(アセンブリ・機械語)
   ↓                        [本作 前編: 何を抽象化したか]
レジスタ・スタック・呼出規約
   ↓                        [本作 後編: 漏れの 5 ケース]
CPU パイプライン・キャッシュ・メモリ階層 ← ここからソースの順序が守られない
```

**高級言語の価値は、上の階段を全部降りなくていいこと**にあります。同時に、**どの段に問題が住んでいるかを当てる能力**は、エンジニアの根幹的なスキルです。今日の事故が「ロジック層の問題」なのか、「キャッシュ層の問題」なのか、「ABI 層の問題」なのかを最初の 5 分で見立てられるかどうかで、デバッグの方向はまったく変わります。

普段の業務では、ほとんどの問題は階段の上の方で解けます。けれど、ごく稀に、本当に下まで降りる必要のある瞬間があります。そのとき「自分はどの階に居て、どこに穴があるか」が分かるようになっていることが、この 2 部作の本当の目的でした。

アセンブラを書ける必要はない。**抽象化の漏れ方を読める語彙**が手元にあれば十分です。
